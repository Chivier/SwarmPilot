# Instance Service API Reference

## Base URL

```
http://<instance_host>:<instance_port>
```

Example: `http://localhost:5000`

## Authentication

Currently, no authentication is required. Future versions may implement token-based authentication.

---

## Model Management APIs

### POST `/model/start`

Start a model/tool on this instance. Each instance can only serve one model at a time.

**Request Body:**
```json
{
  "model_id": "string",           // Unique identifier of the model/tool to start
  "parameters": {                 // Model-specific initialization parameters (optional)
    "key": "value"
  },
  "scheduler_url": "string"       // Target scheduler's url
}
```

**Example Requests:**
```json
{
  "model_id": "easyocr/det",
  "parameters": {}
}
```

```json
{
  "model_id": "simple_mlp",
  "parameters": {
    "layers": [32, 32, 32, 64, 64, 64, 32, 32]
  }
}
```

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Model started successfully",
  "model_id": "string",
  "status": "running"
}
```

**Response (400 Bad Request):**
```json
{
  "success": false,
  "error": "Another model is already running. Stop the current model first."
}
```

**Response (500 Internal Server Error):**
```json
{
  "success": false,
  "error": "Failed to start model: [error details]"
}
```

### GET `/model/stop`

Stop the currently running model on this instance.

**Request Body:**
None

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Model stopped successfully",
  "model_id": "string"
}
```

**Response (400 Bad Request):**
```json
{
  "success": false,
  "error": "No model is currently running"
}
```

---

### POST `/model/restart`

Stop current model, drain from current scheduler, wait for all pending tasks to finish, start a new model, then deregister from current scheduler and register to a new scheduler.

This endpoint performs a graceful model switch operation with the following steps:
1. Stop accepting new tasks from the current scheduler (drain mode)
2. Wait for all pending tasks in the queue to complete
3. Stop the current model container
4. Deregister from the current scheduler
5. Start the new model container
6. Register with the new scheduler (if scheduler_url is provided)

**Request Body:**
```json
{
  "model_id": "string",           // Unique identifier of the model/tool to start
  "parameters": {                 // Model-specific initialization parameters (optional)
    "key": "value"
  },
  "scheduler_url": "string"       // Target scheduler's URL (optional)
}
```

**Example Request:**
```json
{
  "model_id": "easyocr/rec",
  "parameters": {
    "gpu": true
  },
  "scheduler_url": "http://scheduler-2.example.com:8000"
}
```

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Model restarted successfully",
  "old_model_id": "easyocr/det",
  "new_model_id": "easyocr/rec",
  "status": "running",
  "pending_tasks_completed": 5
}
```

**Response (400 Bad Request):**
```json
{
  "success": false,
  "error": "No model is currently running"
}
```

**Response (404 Not Found):**
```json
{
  "success": false,
  "error": "Model not found in registry: invalid_model"
}
```

**Response (500 Internal Server Error):**
```json
{
  "success": false,
  "error": "Failed to restart model: [error details]"
}
```

**Response (503 Service Unavailable):**
```json
{
  "success": false,
  "error": "Restart timeout: pending tasks did not complete within the allowed time"
}
```

**Response (409 Conflict):**
```json
{
  "success": false,
  "error": "A restart operation is already in progress (operation_id: abc-123)"
}
```

---

### GET `/model/restart/status`

Get the status of a model restart operation.

**Query Parameters:**
- `operation_id` (required, string): The operation ID returned by POST `/model/restart`

**Example Request:**
```
GET /model/restart/status?operation_id=abc-123
```

**Response (200 OK) - In Progress:**
```json
{
  "success": true,
  "operation_id": "abc-123",
  "status": "waiting_tasks",
  "old_model_id": "easyocr/det",
  "new_model_id": "easyocr/rec",
  "initiated_at": "2025-10-31T10:30:00Z",
  "completed_at": null,
  "pending_tasks_at_start": 5,
  "pending_tasks_completed": 3,
  "error": null
}
```

**Response (200 OK) - Completed:**
```json
{
  "success": true,
  "operation_id": "abc-123",
  "status": "completed",
  "old_model_id": "easyocr/det",
  "new_model_id": "easyocr/rec",
  "initiated_at": "2025-10-31T10:30:00Z",
  "completed_at": "2025-10-31T10:32:15Z",
  "pending_tasks_at_start": 5,
  "pending_tasks_completed": 5,
  "error": null
}
```

**Response (200 OK) - Failed:**
```json
{
  "success": true,
  "operation_id": "abc-123",
  "status": "failed",
  "old_model_id": "easyocr/det",
  "new_model_id": "invalid_model",
  "initiated_at": "2025-10-31T10:30:00Z",
  "completed_at": "2025-10-31T10:30:05Z",
  "pending_tasks_at_start": 0,
  "pending_tasks_completed": 0,
  "error": "Model not found in registry: invalid_model"
}
```

**Response (404 Not Found):**
```json
{
  "success": false,
  "error": "Restart operation not found: invalid-operation-id"
}
```

**Status Values:**
- `pending` - Restart request received, operation starting
- `draining` - Requesting scheduler to stop sending new tasks
- `waiting_tasks` - Waiting for pending tasks to complete
- `stopping_model` - Stopping the current model container
- `deregistering` - Deregistering from the current scheduler
- `starting_model` - Starting the new model container
- `registering` - Registering with the new scheduler
- `completed` - Restart completed successfully
- `failed` - Restart failed (see `error` field for details)

## Task Management APIs

### POST `/task/submit`

Submit a new task to the task queue. Tasks are processed sequentially in FIFO order.

**Request Body:**
```json
{
  "task_id": "string",            // Unique identifier for this task
  "model_id": "string",           // Model/tool ID to use for this task
  "task_input": {                 // Model-specific input data
    "key": "value"
  }
}
```

**Example Request:**
```json
{
  "task_id": "task-001",
  "model_id": "easyocr/det",
  "task_input": {
    "image_url": "https://example.com/image.png",
    "language": "en"
  }
}
```

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Task submitted successfully",
  "task_id": "string",
  "status": "queued",
  "position": 3                   // Position in the queue
}
```

**Response (400 Bad Request):**
```json
{
  "success": false,
  "error": "Task with this ID already exists"
}
```

**Response (422 Unprocessable Entity):**
```json
{
  "success": false,
  "error": "Model ID does not match the currently running model"
}
```

---

### GET `/task/list`

List all tasks and their current status in the queue.

**Request Body:**
None

**Query Parameters:**
- `status` (optional): Filter by status (`queued`, `running`, `completed`, `failed`)
- `limit` (optional): Maximum number of tasks to return (default: 100)

**Response (200 OK):**
```json
{
  "success": true,
  "total": 5,
  "tasks": [
    {
      "task_id": "string",
      "model_id": "string",
      "status": "completed",       // Status: queued, running, completed, failed
      "submitted_at": "2025-10-31T10:30:00Z",
      "started_at": "2025-10-31T10:30:05Z",
      "completed_at": "2025-10-31T10:30:15Z",
      "result": {                  // Present only if status is "completed"
        "output": "..."
      },
      "error": null                // Present only if status is "failed"
    }
  ]
}
```

### POST `/task/clear`

Clear all tasks from the instance.

This endpoint removes all tasks from the queue and task storage, regardless of their status (queued, completed, failed). Running tasks cannot be cleared - the endpoint will return an error if there are running tasks.

**Use cases:**
- Cleaning up completed/failed task history
- Resetting the instance state before starting new work
- Removing queued tasks that are no longer needed

**Request Body:**
None

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Successfully cleared 15 task(s)",
  "cleared_count": {
    "queued": 3,
    "completed": 10,
    "failed": 2,
    "total": 15
  }
}
```

**Response (400 Bad Request):**
```json
{
  "success": false,
  "error": "Cannot clear tasks while 1 task(s) are running. Wait for running tasks to complete or stop processing first."
}
```

## Usage Examples

### Starting a Model

```bash
curl -X POST http://localhost:5000/model/start \
  -H "Content-Type: application/json" \
  -d '{
    "model_id": "easyocr/det",
    "parameters": {}
  }'
```

### Restarting with a Different Model

```bash
# Initiate the restart operation
RESPONSE=$(curl -X POST http://localhost:5000/model/restart \
  -H "Content-Type: application/json" \
  -d '{
    "model_id": "easyocr/rec",
    "parameters": {
      "gpu": true
    },
    "scheduler_url": "http://scheduler-2.example.com:8000"
  }')

echo $RESPONSE

# Extract operation_id
OPERATION_ID=$(echo $RESPONSE | jq -r '.operation_id')

# Monitor restart progress
while true; do
  STATUS=$(curl -s "http://localhost:5000/model/restart/status?operation_id=$OPERATION_ID" | jq -r '.status')
  echo "Current status: $STATUS"

  if [ "$STATUS" = "completed" ] || [ "$STATUS" = "failed" ]; then
    echo "Restart operation finished with status: $STATUS"
    curl "http://localhost:5000/model/restart/status?operation_id=$OPERATION_ID" | jq
    break
  fi

  sleep 5
done
```

### Submitting a Task

```bash
curl -X POST http://localhost:5000/task/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_id": "task-001",
    "model_id": "easyocr/det",
    "task_input": {
      "image_url": "https://example.com/image.png"
    }
  }'
```

### Checking Task Status

```bash
curl http://localhost:5000/task/task-001
```

### Listing All Tasks

```bash
curl "http://localhost:5000/task/list?status=completed&limit=10"
```

### Clearing All Tasks

```bash
curl -X POST http://localhost:5000/task/clear
```

### Getting Instance Info

```bash
curl http://localhost:5000/info
```

### Health Check

```bash
curl http://localhost:5000/health
```

## Error Response Format

All error responses follow this standard format:

```json
{
  "success": false,
  "error": "Descriptive error message"
}
```

## HTTP Status Codes

| Code | Meaning | Usage |
|------|---------|-------|
| 200 | OK | Request succeeded |
| 400 | Bad Request | Invalid request parameters |
| 404 | Not Found | Resource not found |
| 422 | Unprocessable Entity | Valid request but cannot be processed |
| 500 | Internal Server Error | Server-side error |
| 503 | Service Unavailable | Service is temporarily unavailable |
