# Model Container Specification

## Overview

All models/tools must be containerized as Docker images following this standardized specification to ensure compatibility with the Instance Service.

## Required Components

### 1. Docker Configuration

#### `docker-compose.yaml`
A template file that the instance can modify to inject runtime parameters via environment variables.

**Requirements:**
- Must expose the model's HTTP service port (default: 8000)
- Must define configurable environment variables for model-specific parameters
- Must support resource limit configuration

**Example environment variables:**
- `MODEL_DEVICE`: Computation device (e.g., "cpu", "cuda")
- `BATCH_SIZE`: Inference batch size
- `MODEL_WEIGHTS_PATH`: Path to model weights file

#### `Dockerfile`
Container build instructions following best practices:
- Use appropriate base images (e.g., python:3.11-slim, nvidia/cuda)
- Install uv for dependency management
- Copy application code and dependencies
- Set appropriate working directory
- Configure entry point

---

### 2. Entry Point Script

#### `start.sh`
Shell script serving as the Docker container's entry point.

**Requirements:**
- Must initialize the Python environment using `uv`
- Must start the HTTP service (e.g., FastAPI/Flask application)
- Must handle graceful shutdown signals (SIGTERM, SIGINT)

**Example:**
```bash
#!/bin/bash
set -e

# Initialize uv environment
uv sync

# Start the HTTP service
exec uv run python main.py
```

---

### 3. Python Environment (uv-based)

The model must use [uv](https://github.com/astral-sh/uv) for dependency management.

**Required Files:**
- **`pyproject.toml`**: Project metadata and dependencies
- **`uv.lock`**: Locked dependency versions for reproducibility
- **`main.py`**: Application entry point implementing the HTTP service

**Additional Files:**
- Model-specific Python modules as needed
- Configuration files
- Model weights (if not downloaded at runtime)

---

### 4. Required HTTP Endpoints

Each model container must implement the following standardized endpoints:

#### POST `/inference`
Execute inference with the model using the provided input data.

**Request Body:**
```json
{
  "key": "value"  // Model-specific input fields (Dict[str, Any])
}
```

**Behavior:**
- The `task_input` field from the `Task` object is passed directly to this endpoint without modification
- The endpoint must validate the input according to the model's requirements
- Must return results in a consistent JSON format

**Response (200 OK):**
```json
{
  "success": true,
  "result": {
    "output": "..."  // Model-specific output data
  },
  "execution_time": 1.234,      // Execution time in seconds
  "start_timestamp": 1111111    // Unix timestamp when inference started
}
```

**Response (400 Bad Request):**
```json
{
  "success": false,
  "error": "Invalid input: [detailed error message]"
}
```

**Response (500 Internal Server Error):**
```json
{
  "success": false,
  "error": "Inference failed: [detailed error message]"
}
```

#### GET `/health`
Health check endpoint for monitoring.

**Response (200 OK):**
```json
{
  "status": "healthy",
  "model_loaded": true
}
```

**Response (503 Service Unavailable):**
```json
{
  "status": "unhealthy",
  "error": "Model not loaded or initialization failed"
}
```

## Directory Structure

### Standard Model Container Layout

```
model-container/
├── docker-compose.yaml      # Docker composition template
├── Dockerfile               # Container build instructions
├── start.sh                 # Entry point script
├── pyproject.toml           # Python project config (uv)
├── uv.lock                  # Locked dependencies
├── main.py                  # FastAPI/Flask application
├── model/                   # Model implementation
│   ├── __init__.py
│   ├── inference.py         # Core inference logic
│   └── utils.py
└── weights/                 # Model weights (optional)
    └── model.pth
```

## Environment Variables

### Standard Variables (Injected by Instance)

| Variable | Type | Description |
|----------|------|-------------|
| `MODEL_ID` | string | Unique model identifier |
| `INSTANCE_ID` | string | Instance identifier |
| `LOG_LEVEL` | string | Logging level (INFO, DEBUG, ERROR) |

### Custom Variables (From Request Parameters)

Custom parameters from the `POST /model/start` request are converted to environment variables:
- Converted to uppercase
- Prefixed with `MODEL_`
- Complex types (arrays, objects) are JSON-encoded

**Example:**
```json
// Request parameters:
{
  "layers": [32, 64],
  "device": "cuda"
}

// Results in environment variables:
MODEL_LAYERS="[32, 64]"
MODEL_DEVICE="cuda"
```

## Example Implementation

### Minimal FastAPI Application (`main.py`)

```python
import os
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import Dict, Any
import time

app = FastAPI()

# Load model on startup
model = None

@app.on_event("startup")
async def load_model():
    global model
    model_id = os.getenv("MODEL_ID")
    # Load your model here
    model = YourModel.load(model_id)

@app.post("/inference")
async def inference(input_data: Dict[str, Any]):
    if model is None:
        raise HTTPException(status_code=503, detail="Model not loaded")

    try:
        start_time = time.time()
        start_timestamp = int(start_time)

        result = model.predict(input_data)

        execution_time = time.time() - start_time

        return {
            "success": True,
            "result": result,
            "execution_time": execution_time,
            "start_timestamp": start_timestamp
        }
    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Inference failed: {str(e)}")

@app.get("/health")
async def health():
    if model is None:
        return {"status": "unhealthy", "model_loaded": False}, 503
    return {"status": "healthy", "model_loaded": True}
```

## Best Practices

1. **Resource Management**
   - Implement proper model cleanup on shutdown
   - Use connection pooling for external resources
   - Set appropriate timeout values

2. **Error Handling**
   - Validate all input data before processing
   - Return descriptive error messages
   - Log errors for debugging

3. **Performance**
   - Warm up models during startup
   - Implement batch processing when appropriate
   - Cache frequently used resources

4. **Security**
   - Validate input data types and ranges
   - Sanitize file paths and external inputs
   - Use environment variables for sensitive configuration

---

## Validation Checklist

Before deploying a model container, verify:

- [ ] `docker-compose.yaml` exists and is valid
- [ ] `Dockerfile` builds successfully
- [ ] `start.sh` is executable and handles shutdown signals
- [ ] `pyproject.toml` and `uv.lock` are present
- [ ] `/inference` endpoint returns correct response format
- [ ] `/health` endpoint responds correctly
- [ ] Environment variables are properly consumed
- [ ] Model loads successfully on container startup
- [ ] Resource limits are appropriately configured
