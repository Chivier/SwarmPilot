# Instance Service Architecture

## Overview

The Instance Service is the minimal execution unit and scheduling unit in the SwarmPilot project. Each instance serves a single model at a time and provides task queue management for processing inference requests sequentially.

## Core Features

### 1. Task Queue Management
- Accept multiple tasks simultaneously
- Process tasks sequentially in FIFO (First-In-First-Out) order
- Track task status throughout execution lifecycle
- Support task cancellation for queued tasks

### 2. Dynamic Model Management
- Each instance serves only one model/tool at a time
- Support hot-swapping between different models via API
- Docker-based model isolation and resource management
- Automatic model container lifecycle management

## Architecture Components

### Instance Core
The main service that:
- Exposes REST API endpoints for management and task submission
- Manages the task queue and execution flow
- Handles model container lifecycle (start/stop)
- Maintains instance state and health status

### Model Container
Docker containers that:
- Run specific ML models or processing tools
- Expose standardized `/inference` endpoint
- Use uv-based Python environment management
- Handle graceful shutdown and resource cleanup

### Task Queue
In-memory or persistent queue that:
- Stores submitted tasks in FIFO order
- Tracks task states: `queued`, `running`, `completed`, `failed`
- Provides task query and management capabilities

## Task Lifecycle

```
┌─────────────┐
│  Submitted  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Queued    │ ◄── Task waiting in queue
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Running   │ ◄── Task being processed by model
└──────┬──────┘
       │
       ├─────────────┐
       ▼             ▼
┌─────────────┐ ┌─────────────┐
│  Completed  │ │   Failed    │
└─────────────┘ └─────────────┘
```

## Port Allocation Strategy

The instance uses a predictable port allocation scheme:

- **Instance API**: `<instance_port>` (e.g., 5000)
- **Model Container**: `<instance_port> + 1000` (e.g., 6000)

This allows:
- Multiple instances to run on the same host
- Clear port separation between instance and model services
- Easy proxy routing from instance to model

## Instance States

| State | Description |
|-------|-------------|
| `idle` | Instance running, no model loaded |
| `running` | Model loaded and accepting tasks |
| `busy` | Model processing a task |
| `error` | Instance or model in error state |

## Scaling Considerations

### Horizontal Scaling
- Deploy multiple instances with different `instance_id`
- Each instance operates independently
- External scheduler (e.g., SwarmPilot Scheduler) distributes tasks

### Resource Isolation
- Each model container has dedicated resource limits (CPU, memory, GPU)
- Docker provides process isolation and resource constraints
- Prevents resource contention between models
