## Overview

The Predictor can operate in two modes:

1. **Expect & Error (`"expect_error"`)**: Returns a single expected runtime value with an error margin
2. **Quantile (`"quantile"`)**: Returns a distribution of runtimes at specified percentiles

Both types use the same underlying MLP (Multi-Layer Perceptron) models but are trained and optimized differently.

---

## Expect & Error

### Description

The `"expect_error"` prediction type provides a simple, single-value estimate of task runtime along with an error margin representing uncertainty.

### Output Format

```json
{
  "expected_runtime_ms": 125.5,
  "error_margin_ms": 10.2
}
```

### Field Meanings

| Field | Description |
|-------|-------------|
| `expected_runtime_ms` | The mean/average predicted runtime in milliseconds |
| `error_margin_ms` | Uncertainty margin (typically ±1 standard deviation) |

### Interpretation

The predicted runtime follows approximately a normal distribution:
- **68% confidence interval**: `expected_runtime_ms ± error_margin_ms`
- **Example**: If `expected_runtime_ms = 100.0` and `error_margin_ms = 10.0`:
  - 68% of tasks will complete between 90ms and 110ms
  - 95% of tasks will complete between 80ms and 120ms (±2σ)

### Use Cases

**Best for**:
- Simple scheduling decisions based on average runtime
- Resource allocation and capacity planning
- Quick estimations when tail latencies are not critical
- General-purpose scheduling algorithms

**Example scenarios**:
- Load balancing across multiple instances
- Basic queue management
- Cost estimation for task execution
- Simple deadline checking

### Training

Train an expect_error model:
```bash
curl -X POST http://localhost:8000/train \
  -H "Content-Type: application/json" \
  -d '{
    "model_id": "my-model",
    "platform_info": {...},
    "prediction_type": "expect_error",
    "features_list": [...]
  }'
```

No special configuration required.

### Example Usage

**Scheduling Decision**:
```python
# Get prediction
prediction = predictor.predict(model_id, platform, features)
expected = prediction["result"]["expected_runtime_ms"]
error = prediction["result"]["error_margin_ms"]

# Simple scheduling: use expected runtime
if expected < deadline_ms:
    schedule_task()

# Conservative scheduling: add buffer
if expected + error < deadline_ms:
    schedule_task()
```

## Quantile

### Description

The `"quantile"` prediction type provides a full distribution of runtimes at specified percentiles, enabling sophisticated latency-aware scheduling.

### Output Format

```json
{
  "quantiles": {
    "0.5": 120.0,
    "0.9": 150.0,
    "0.95": 180.0,
    "0.99": 250.0
  }
}
```

### Field Meanings

Each key-value pair represents a percentile:

| Quantile | Meaning |
|----------|---------|
| `0.5` | **Median**: 50% of tasks complete faster than this value |
| `0.9` | **P90**: 90% of tasks complete faster than this value |
| `0.95` | **P95**: 95% of tasks complete faster than this value |
| `0.99` | **P99**: 99% of tasks complete faster than this value |

### Interpretation

**Example**: `{"0.95": 180.0}`
- 95% of tasks will complete in ≤180ms
- 5% of tasks will take >180ms (tail latencies)

**Visualizing the distribution**:
```
Runtime (ms)
    │
250 ┤                              •──── P99 (1% slower)
200 ┤                       •────── P95 (5% slower)
150 ┤                •────── P90 (10% slower)
120 ┤         •────── P50 (median)
    └─────────────────────────────────────
              Tasks (percentile)
```

### Use Cases

**Best for**:
- Latency-sensitive applications with SLA requirements
- Tail latency optimization
- Advanced scheduling algorithms
- Understanding runtime variance and distribution shape

**Example scenarios**:
- SLA-based scheduling ("99% of tasks must complete within X ms")
- P99 latency guarantees for real-time systems
- Identifying and avoiding high-variance workloads
- Detailed performance analysis

### Training with Custom Quantiles

Specify which quantiles to predict:

```bash
curl -X POST http://localhost:8000/train \
  -H "Content-Type: application/json" \
  -d '{
    "model_id": "my-model",
    "platform_info": {...},
    "prediction_type": "quantile",
    "features_list": [...],
    "training_config": {
      "quantiles": [0.5, 0.9, 0.95, 0.99]
    }
  }'
```

### Common Quantile Configurations

| Configuration | Use Case |
|--------------|----------|
| `[0.5]` | Median only (fastest, for simple decisions) |
| `[0.5, 0.9, 0.95]` | Median with tail latencies |
| `[0.5, 0.95, 0.99]` | SLA-focused (common SLAs) |
| `[0.25, 0.5, 0.75, 0.9, 0.95, 0.99]` | Full distribution analysis |

### Example Usage

**SLA-Based Scheduling**:
```python
# SLA requirement: 95% of tasks must complete within 200ms
prediction = predictor.predict(model_id, platform, features)
p95_runtime = prediction["result"]["quantiles"]["0.95"]

if p95_runtime <= 200:
    # SLA will be met
    schedule_task()
else:
    # SLA will be violated
    find_faster_instance()
```

**Tail Latency Optimization**:
```python
# Choose instance with best P99 latency
instances = get_available_instances()
predictions = [
    predictor.predict(model_id, instance.platform, features)
    for instance in instances
]

best_instance = min(
    predictions,
    key=lambda p: p["result"]["quantiles"]["0.99"]
)
```

## Comparison

### Quick Comparison Table

| Feature | expect_error | quantile |
|---------|-------------|----------|
| **Output Complexity** | Simple (2 values) | Complex (N quantiles) |
| **Tail Latency Info** | Limited (via error) | Explicit percentiles |
| **Prediction Speed** | Fast | Slightly slower |
| **Training Config** | None | Configurable quantiles |
| **Best For** | General scheduling | SLA-aware scheduling |
| **Output Size** | ~40 bytes | ~100-200 bytes |
| **Scheduler Complexity** | Low | Medium to High |

### Detailed Comparison

#### Information Richness

**expect_error**:
- Provides mean and standard deviation
- Assumes normal distribution (may not be accurate for skewed distributions)
- Cannot distinguish between symmetric and skewed distributions

**quantile**:
- Provides actual distribution shape
- No distribution assumptions
- Can reveal skewness and heavy tails

#### Scheduling Flexibility

**expect_error**:
```python
# Limited to simple checks
if expected_runtime < deadline:
    schedule()
```

**quantile**:
```python
# Can optimize for specific percentiles
if quantiles["0.99"] < strict_deadline:
    schedule()  # 99% guarantee
elif quantiles["0.95"] < relaxed_deadline:
    schedule()  # 95% guarantee
```

#### Performance Impact

**expect_error**:
- Faster training (simpler model)
- Faster prediction (single output)
- Smaller model size

**quantile**:
- Slower training (more complex model)
- Slightly slower prediction (multiple outputs)
- Larger model size

## Choosing the Right Type

### Use `"expect_error"` When:

✓ You need simple, fast predictions
✓ Average runtime is sufficient for your scheduling logic
✓ Tail latencies are not critical to your application
✓ You want minimal complexity in your scheduler
✓ You're implementing basic load balancing
✓ Cost estimation is more important than latency guarantees

**Example**: Batch processing system where throughput matters more than individual task latency.

### Use `"quantile"` When:

✓ You have SLA requirements (e.g., "P99 < 200ms")
✓ Tail latencies significantly impact user experience
✓ You need to make guarantees about runtime distribution
✓ Your scheduler can optimize for specific percentiles
✓ You're implementing latency-sensitive scheduling
✓ Understanding variance is important

**Example**: Real-time API serving with strict latency SLAs.

### Decision Flow

```
Do you have SLA requirements?
├─ Yes → Use quantile
└─ No
   └─ Are tail latencies critical?
      ├─ Yes → Use quantile
      └─ No
         └─ Do you need simple predictions?
            ├─ Yes → Use expect_error
            └─ No
               └─ Do you need distribution info?
                  ├─ Yes → Use quantile
                  └─ No → Use expect_error
```

## Technical Details

### Model Training

Both types use Multi-Layer Perceptron (MLP) models but with different loss functions:

**expect_error**:
- Loss function: Mean Squared Error (MSE)
- Outputs: 2 neurons (mean, stddev)
- Training objective: Minimize prediction error on mean and variance

**quantile**:
- Loss function: Pinball loss (quantile regression)
- Outputs: N neurons (one per quantile)
- Training objective: Minimize quantile prediction errors

### Prediction Behavior

**expect_error**:
1. Forward pass through MLP
2. Output layer produces mean and stddev
3. Format as `{expected_runtime_ms, error_margin_ms}`

**quantile**:
1. Forward pass through MLP
2. Output layer produces quantile values
3. Format as `{quantiles: {q1: v1, q2: v2, ...}}`

### Experiment Mode Behavior

**expect_error**:
```json
{
  "expected_runtime_ms": <exp_runtime>,
  "error_margin_ms": <exp_runtime * 0.05>
}
```

**quantile**:
```json
{
  "quantiles": {
    "0.5": <exp_runtime>,
    "0.9": <exp_runtime * 1.05>,
    "0.95": <exp_runtime * 1.075>,
    "0.99": <exp_runtime * 1.12>
  }
}
```

## Real-World Examples

### Example 1: E-commerce Platform

**Scenario**: Image processing service for product photos

**Requirements**:
- Must process images within 500ms for 95% of requests
- Occasional slow requests (>1s) are acceptable

**Solution**: Use `"quantile"` with `[0.5, 0.95, 0.99]`

```python
prediction = predict(image_params)
if prediction["quantiles"]["0.95"] < 500:
    # 95% SLA will be met
    process_on_this_instance()
else:
    # Find faster instance
    try_different_instance()
```

### Example 2: Batch Video Processing

**Scenario**: Overnight batch processing of videos

**Requirements**:
- Process all videos before morning
- Individual video latency doesn't matter

**Solution**: Use `"expect_error"`

```python
prediction = predict(video_params)
total_expected_time = sum([
    pred["expected_runtime_ms"] for pred in predictions
])
if total_expected_time < time_until_morning:
    start_batch()
```

### Example 3: ML Model Serving

**Scenario**: Real-time inference API

**Requirements**:
- P99 latency < 100ms
- P50 latency < 50ms

**Solution**: Use `"quantile"` with `[0.5, 0.9, 0.95, 0.99]`

```python
predictions = [predict(model, instance) for instance in instances]
best = min(predictions, key=lambda p: (
    p["quantiles"]["0.99"] < 100,  # P99 requirement
    p["quantiles"]["0.5"]           # Optimize P50
))
```

## Summary

- **expect_error**: Simple, fast, good for average-case scheduling
- **quantile**: Detailed, flexible, essential for SLA-aware scheduling
- **Choose based on**: Your SLA requirements, tail latency sensitivity, and scheduler complexity

For most production systems with latency requirements, **quantile** is recommended despite the additional complexity.
