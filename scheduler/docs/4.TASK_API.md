# Task Management API

This document describes the API endpoints for submitting, tracking, and querying tasks.

> **See Also**:
> - [Data Models](2.DATA_MODELS.md) for request/response model definitions
> - [WebSocket API](7.WEBSOCKET_API.md) for real-time task result notifications

---

## Overview

Tasks represent computational work to be executed on instances. The scheduler receives tasks, predicts runtime on available instances, selects the best instance using the configured scheduling strategy, and dispatches the task for execution.

**Task Lifecycle:**
1. **PENDING** - Task submitted, background scheduler is assigning instance
2. **RUNNING** - Task dispatched to instance and executing
3. **COMPLETED** - Task finished successfully with result
4. **FAILED** - Task failed with error message

**⚠️ 重要变更（v2.0+）：后台调度模式**

从v2.0开始，任务调度在后台异步执行：
- API立即返回（~5ms响应时间）
- 任务初始状态为 `PENDING`，`assigned_instance` 为空
- 后台调度器异步分配实例（通常<100ms）
- 使用WebSocket或轮询获取最终分配结果

参见：[后台调度器文档](10.BACKGROUND_SCHEDULER.md)

## Endpoints

### POST `/task/submit`

Submit a new task to the scheduler for execution.

**Request Body:**
```json
{
  "task_id": "task-ocr-001",
  "model_id": "easyocr/dec",
  "task_input": {
    "image": "base64_encoded_image_data"
  },
  "metadata": {
    "height": 512,
    "width": 512
  }
}
```

**Field Descriptions:**
- `task_id`: Unique identifier for this task (must be unique across all tasks)
- `model_id`: Model or tool to use (must match a registered instance's model_id)
- `task_input`: Flexible input data passed to the model (structure depends on model requirements)
- `metadata`: Additional information for runtime prediction (e.g., image dimensions, text length)

**Success Response (200 OK):**
```json
{
  "success": true,
  "message": "Task submitted successfully and is being scheduled in background",
  "task": {
    "task_id": "task-ocr-001",
    "status": "pending",
    "assigned_instance": "",
    "submitted_at": "2025-10-31T12:34:56.789Z"
  }
}
```

**Response Fields:**
- `assigned_instance`: **Empty initially**, filled by background scheduler within ~100ms
- `status`: Initially "pending", transitions to "running" when dispatched
- `submitted_at`: ISO 8601 timestamp with timezone

**⚠️ 后台调度行为（v2.0+）：**
- API立即返回（~5ms），无需等待调度完成
- `assigned_instance` 初始为空字符串 `""`
- 后台调度器异步完成：
  1. 调用predictor获取预测（10-50ms）
  2. 选择最优实例（1-5ms）
  3. 更新队列信息（5-10ms）
  4. 分配实例并分发任务
- 使用 **WebSocket订阅** 或 **轮询** `/task/query` 获取最终分配结果

**Error Responses:**

**400 Bad Request** - Task ID already exists:
```json
{
  "success": false,
  "error": "Task with this ID already exists"
}
```

**404 Not Found** - No instances available for model:
```json
{
  "success": false,
  "error": "No available instance for model_id: easyocr/dec"
}
```

**⚠️ v2.0+ 错误处理变化：**

某些错误现在在后台处理，不会直接返回给API调用：
- ~~503 Predictor service unavailable~~ → 任务标记为 FAILED (后台)
- ~~504 Scheduling timeout~~ → 任务标记为 FAILED (后台)

查看任务状态以获取后台错误：
```bash
GET /task/query?task_id=task-ocr-001

# 响应：
{
  "task": {
    "status": "failed",
    "error": "Scheduling failed: Predictor service unavailable"
  }
}
```

**How Task Submission Works (v2.0+ 后台调度):**

**API阶段（同步，~5ms）：**
1. 验证请求，检查重复task_id
2. 快速检查是否有可用实例
3. 创建任务记录（status=PENDING, assigned_instance=""）
4. 提交后台调度任务
5. **立即返回**

**后台调度阶段（异步，~50-100ms）：**
1. 查询可用实例列表
2. 调用predictor服务获取运行时预测
3. 调度策略根据预测和队列状态选择最优实例
4. 更新队列信息（蒙特卡洛采样）
5. 分配实例到任务
6. 分发任务到实例执行

**客户端推荐做法：**

**方式1：WebSocket订阅（推荐）**
```python
# 1. 建立WebSocket连接并订阅
ws.send({"type": "subscribe", "task_ids": ["task-ocr-001"]})

# 2. 提交任务
POST /task/submit

# 3. 等待WebSocket推送结果（无需轮询）
ws.receive()  # 自动收到完成通知
```

**方式2：轮询**
```python
# 1. 提交任务
response = POST /task/submit
task_id = response["task"]["task_id"]

# 2. 轮询直到assigned_instance不为空
while True:
    task = GET /task/query?task_id={task_id}
    if task["assigned_instance"]:
        break
    time.sleep(0.1)  # 100ms
```

**Example:**
```bash
curl -X POST http://localhost:8000/task/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_id": "task-ocr-001",
    "model_id": "easyocr/dec",
    "task_input": {
      "image": "base64_encoded_image_data"
    },
    "metadata": {
      "height": 512,
      "width": 512
    }
  }'
```

### GET `/task/list`

List tasks with optional filtering and pagination.

**Query Parameters:**
- `status` (optional, enum): Filter by task status (`pending`, `running`, `completed`, `failed`)
- `model_id` (optional, string): Filter by model ID
- `instance_id` (optional, string): Filter by assigned instance ID
- `limit` (optional, integer, default=100, range=1-1000): Maximum tasks to return
- `offset` (optional, integer, default=0): Pagination offset

**Example Requests:**
```
GET /task/list
GET /task/list?status=completed
GET /task/list?model_id=easyocr/dec&limit=50
GET /task/list?instance_id=instance-gpu-0&offset=100&limit=50
```

**Success Response (200 OK):**
```json
{
  "success": true,
  "count": 2,
  "total": 150,
  "offset": 0,
  "limit": 100,
  "tasks": [
    {
      "task_id": "task-ocr-001",
      "model_id": "easyocr/dec",
      "status": "completed",
      "assigned_instance": "instance-gpu-0",
      "submitted_at": "2025-10-31T12:34:56.789Z",
      "completed_at": "2025-10-31T12:35:01.234Z"
    },
    {
      "task_id": "task-ocr-002",
      "model_id": "easyocr/dec",
      "status": "running",
      "assigned_instance": "instance-gpu-1",
      "submitted_at": "2025-10-31T12:35:10.456Z",
      "completed_at": null
    }
  ]
}
```

**Response Fields:**
- `count`: Number of tasks in current page
- `total`: Total tasks matching filter criteria
- `offset`: Current pagination offset
- `limit`: Current pagination limit
- `tasks`: Array of task summaries (ordered by submission time, newest first)
- `completed_at`: Present only for completed/failed tasks

**Pagination Example:**

To retrieve all tasks in pages of 50:
```bash
# Page 1
curl "http://localhost:8000/task/list?limit=50&offset=0"

# Page 2
curl "http://localhost:8000/task/list?limit=50&offset=50"

# Page 3
curl "http://localhost:8000/task/list?limit=50&offset=100"
```

### POST `/task/clear`

Clear all tasks from the scheduler and all registered instances.

**Warning**: This operation removes **all** task records permanently and cannot be undone. Use with caution, especially in production environments.

**Request:**
```
POST /task/clear
```

No request body required.

**Success Response (200 OK):**
```json
{
  "success": true,
  "message": "Successfully cleared 42 scheduler task(s) and tasks from 3/3 instance(s)",
  "cleared_count": 42
}
```

**Response Fields:**
- `cleared_count`: Number of tasks that were removed from the scheduler registry
- `message`: Human-readable confirmation message including instance clear status

**What This Endpoint Does:**

This endpoint performs a comprehensive cleanup across the entire system:

1. **Clears scheduler task registry** - Removes all task records from the scheduler
2. **Calls `/task/clear` on all instances** - Sends HTTP POST requests to each registered instance to clear their task queues
3. **Resets pending task counters** - Resets the `pending_tasks` counter for all instances to zero

**Use Cases:**
- Cleaning up test data during development
- Resetting system state between benchmark runs
- Clearing stale tasks during maintenance windows
- Preparing for strategy switches

**Example:**
```bash
curl -X POST http://localhost:8000/task/clear
```

**Important Notes:**
- This endpoint **will call** `/task/clear` on each registered instance
- Instance clearing continues even if some instances fail to respond
- Tasks currently running on instances **cannot** be cleared (instances will reject the request)
- If any instance has running tasks, that instance's tasks will not be cleared
- All task history (pending, running, completed, failed) is cleared from scheduler
- WebSocket subscribers will not be notified of cleared tasks
- Failed instance clears are logged but do not fail the overall operation

**Instance Clearing Behavior:**

For each registered instance:
- **Success**: Instance clears its task queue and returns the count of cleared tasks
- **Failure (instance has running tasks)**: Instance returns error, tasks not cleared from that instance
- **Failure (network/timeout)**: Logged as warning, operation continues with other instances

**Recommended Workflow for Safe Clearing:**
```bash
# 1. Check current task count on scheduler
curl "http://localhost:8000/task/list" | jq '.total'

# 2. Review running tasks (must be 0 for instances to clear)
curl "http://localhost:8000/task/list?status=running"

# 3. Optionally check each instance's task status
curl "http://instance-1:5000/task/list?status=running"

# 4. Clear all tasks (scheduler + all instances)
curl -X POST "http://localhost:8000/task/clear"

# 5. Verify clearing on scheduler
curl "http://localhost:8000/task/list" | jq '.total'  # Should return 0

# 6. Optionally verify clearing on instances
curl "http://instance-1:5000/task/list" | jq '.total'  # Should return 0
```

**Error Scenarios:**

If instances have running tasks, you'll see warnings in the scheduler logs:
```
[WARNING] Failed to clear tasks from instance instance-gpu-0:
          Cannot clear tasks while 1 task(s) are running
```

The scheduler's tasks will still be cleared, but that instance's tasks remain until they complete.

### POST `/task/update_metadata`

Update metadata for multiple tasks and trigger queue recalculation for tasks that are currently in an instance queue.

**Description:**

This endpoint allows batch updating of task metadata. When a task is currently assigned to an instance (status is PENDING or RUNNING), the system will:
1. Remove the old predicted execution time from the instance queue
2. Re-predict execution time with the new metadata
3. Add the new predicted time to the instance queue
4. Update the task's prediction fields

This is useful for scenarios where task characteristics change after submission (e.g., updated input parameters that affect execution time).

**Request Body:**
```json
{
  "updates": [
    {
      "task_id": "task-001",
      "metadata": {
        "height": 1024,
        "width": 1024,
        "quality": "high"
      }
    },
    {
      "task_id": "task-002",
      "metadata": {
        "height": 512,
        "width": 512
      }
    }
  ]
}
```

**Field Descriptions:**
- `updates` (required, array): List of task metadata updates
  - `task_id` (required, string): ID of the task to update
  - `metadata` (required, object): New metadata dictionary (completely replaces existing metadata)

**Success Response (200 OK):**
```json
{
  "success": true,
  "message": "Updated 2/2 tasks (0 skipped, 0 failed)",
  "total": 2,
  "succeeded": 2,
  "failed": 0,
  "skipped": 0,
  "results": [
    {
      "task_id": "task-001",
      "success": true,
      "message": "Metadata updated with queue recalculation",
      "queue_updated": true,
      "old_prediction_ms": 100.0,
      "new_prediction_ms": 250.0
    },
    {
      "task_id": "task-002",
      "success": true,
      "message": "Metadata updated (task not in queue)",
      "queue_updated": false,
      "old_prediction_ms": null,
      "new_prediction_ms": null
    }
  ]
}
```

**Response Fields:**
- `success`: Overall success (true if no failures)
- `message`: Summary message
- `total`: Total number of updates requested
- `succeeded`: Number of successful updates
- `failed`: Number of failed updates
- `skipped`: Number of skipped updates (COMPLETED/FAILED tasks)
- `results`: Array of per-task results
  - `task_id`: Task ID
  - `success`: Whether this specific update succeeded
  - `message`: Status message explaining what happened
  - `queue_updated`: Whether the instance queue was recalculated
  - `old_prediction_ms`: Previous predicted execution time (null if no prediction)
  - `new_prediction_ms`: New predicted execution time (null if no queue update)

**Behavior Notes:**

1. **Metadata Replacement**: New metadata **completely replaces** existing metadata (no merge). If the old metadata had keys not present in the new metadata, those keys are removed.

2. **COMPLETED/FAILED Tasks**: Tasks with status `COMPLETED` or `FAILED` are silently skipped (no error, counted in `skipped`). Queue recalculation is not needed for completed tasks.

3. **Rollback on Prediction Failure**: If the predictor service fails during re-prediction, the metadata change is rolled back and the original queue values are restored. This ensures atomicity.

4. **Queue Update Flow** (重预测 → 移除旧值 → 向队列中叠加新值):
   - Subtract old prediction from queue
   - Update metadata in task registry
   - Call predictor with new metadata
   - Add new prediction to queue
   - Update task's prediction fields

5. **Instance Not Found**: If the task's assigned instance no longer exists, metadata is still updated but queue recalculation is skipped with a warning.

**Error Scenarios:**

**Task Not Found:**
```json
{
  "results": [
    {
      "task_id": "nonexistent-task",
      "success": false,
      "message": "Task not found",
      "queue_updated": false
    }
  ]
}
```

**Prediction Failure (with rollback):**
```json
{
  "results": [
    {
      "task_id": "task-001",
      "success": false,
      "message": "Prediction failed, rolled back: Predictor service unavailable",
      "queue_updated": false,
      "old_prediction_ms": 100.0
    }
  ]
}
```

**Validation Error (422 Unprocessable Entity):**
```json
{
  "detail": [
    {
      "loc": ["body", "updates", 0, "metadata"],
      "msg": "field required",
      "type": "value_error.missing"
    }
  ]
}
```

**Example:**
```bash
curl -X POST http://localhost:8000/task/update_metadata \
  -H "Content-Type: application/json" \
  -d '{
    "updates": [
      {
        "task_id": "task-ocr-001",
        "metadata": {
          "height": 1024,
          "width": 1024,
          "format": "png"
        }
      }
    ]
  }'
```

**Python Example:**
```python
import httpx

async def update_task_metadata(scheduler_url: str, task_updates: list):
    """Update metadata for multiple tasks."""
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"{scheduler_url}/task/update_metadata",
            json={"updates": task_updates},
            timeout=30.0
        )
        response.raise_for_status()
        return response.json()

# Usage
updates = [
    {"task_id": "task-001", "metadata": {"height": 1024, "width": 1024}},
    {"task_id": "task-002", "metadata": {"height": 512, "width": 512}},
]
result = await update_task_metadata("http://localhost:8000", updates)

# Check results
for task_result in result["results"]:
    if task_result["success"]:
        print(f"Task {task_result['task_id']}: {task_result['message']}")
        if task_result["queue_updated"]:
            print(f"  Prediction: {task_result['old_prediction_ms']}ms -> {task_result['new_prediction_ms']}ms")
    else:
        print(f"Task {task_result['task_id']} failed: {task_result['message']}")
```

**Use Cases:**

1. **Dynamic Task Parameters**: When task input parameters change after submission but before execution.

2. **Priority Adjustment**: Updating metadata that affects prediction (e.g., quality settings, batch size).

3. **Workflow Orchestration**: In multi-stage workflows where intermediate results affect subsequent task characteristics.

4. **Error Recovery**: Correcting metadata that was incorrectly set during initial submission.

**Performance Considerations:**

- Each task with queue update requires a predictor API call (~10-50ms)
- Batch updates are processed sequentially (not parallelized)
- For large batch updates (100+ tasks), consider splitting into smaller batches
- Tasks not in queue (no assigned instance) are fast (~1ms each)

### POST `/task/repredict`

Trigger re-prediction for all pending/running tasks and update queue information **without modifying metadata**.

**Description:**

This endpoint re-predicts all tasks that are currently assigned to an instance (status PENDING or RUNNING). Unlike `/task/update_metadata`, this endpoint does not change any task metadata - it only refreshes predictions using the existing metadata.

**Use case:** When the predictor model has been updated or retrained, use this endpoint to refresh all queue predictions to reflect the new model's estimates.

**Request Body:** None required

**Response Body:**

```json
{
  "success": true,
  "message": "Re-predicted 5/5 eligible tasks (3 skipped, 0 failed)",
  "total_tasks": 8,
  "eligible_tasks": 5,
  "repredicted": 5,
  "failed": 0,
  "skipped": 3
}
```

**Response Fields:**

| Field | Type | Description |
|-------|------|-------------|
| `success` | `bool` | `true` if no failures occurred |
| `message` | `str` | Human-readable summary |
| `total_tasks` | `int` | Total number of tasks in registry |
| `eligible_tasks` | `int` | Tasks eligible for re-prediction (PENDING/RUNNING in queue) |
| `repredicted` | `int` | Successfully re-predicted tasks |
| `failed` | `int` | Tasks that failed re-prediction |
| `skipped` | `int` | Skipped tasks (COMPLETED/FAILED/not in queue) |

**Behavior Notes:**

1. **No Metadata Changes**: Task metadata is preserved unchanged. Only prediction values are updated.

2. **Eligible Tasks**: Only tasks with status PENDING or RUNNING that have an assigned instance are re-predicted.

3. **Skipped Tasks**:
   - COMPLETED tasks: Already finished, no queue impact
   - FAILED tasks: Already finished, no queue impact
   - Tasks without assigned_instance: Not in any queue

4. **Error Handling**: If prediction fails for a task:
   - Queue values are restored to original state
   - Task is counted in `failed` count
   - Processing continues with other tasks

5. **Queue Update Flow** (for each eligible task):
   ```
   1. Subtract old prediction from instance queue
   2. Call predictor with existing metadata
   3. Add new prediction to instance queue
   4. Update task's prediction fields
   ```

**Error Response:**

```json
{
  "success": false,
  "message": "Re-predicted 3/5 eligible tasks (0 skipped, 2 failed)",
  "total_tasks": 5,
  "eligible_tasks": 5,
  "repredicted": 3,
  "failed": 2,
  "skipped": 0
}
```

**Example Usage:**

**curl:**
```bash
curl -X POST http://localhost:8000/task/repredict \
  -H "Content-Type: application/json"
```

**Python:**
```python
import requests

scheduler_url = "http://localhost:8000"

# Trigger re-prediction for all eligible tasks
response = requests.post(f"{scheduler_url}/task/repredict")
result = response.json()

if result["success"]:
    print(f"Successfully re-predicted {result['repredicted']} tasks")
    print(f"Skipped: {result['skipped']}, Failed: {result['failed']}")
else:
    print(f"Re-prediction completed with failures: {result['message']}")
```

**Use Cases:**

1. **Predictor Model Update**: After retraining or updating the prediction model, refresh all queue estimates.

2. **Configuration Change**: When predictor parameters change (e.g., different hardware profiles).

3. **Accuracy Refresh**: Periodically refresh predictions to account for model drift or improved estimates.

4. **System Recovery**: After predictor service restart, ensure queue values are consistent.

**Performance Considerations:**

- Each eligible task requires a predictor API call (~10-50ms)
- Tasks are processed sequentially
- For systems with many pending tasks (1000+), this operation may take several seconds
- Operation does not block task submission or execution

| Field | Type | Description |
|-------|------|-------------|
| `success` | `bool` | Always `true` for successful requests |
| `count` | `int` | Number of tasks in this response (after pagination) |
| `total` | `int` | Total number of tasks matching the filter (before pagination) |
| `tasks` | `list` | List of task scheduling information |

**Task Fields:**

| Field | Type | Description |
|-------|------|-------------|
| `task_id` | `str` | Unique task identifier |
| `model_id` | `str` | Model/tool the task uses |
| `status` | `str` | Current task status |
| `assigned_instance` | `str` | Instance ID the task was scheduled to |
| `submitted_at` | `str` | ISO 8601 timestamp when task was submitted |

**Behavior Notes:**

1. **Empty `assigned_instance`**: Tasks still in the central queue (not yet scheduled) will have an empty `assigned_instance` value.

2. **Sorting**: Tasks are sorted by submission time (newest first).

3. **Lightweight Response**: Unlike `/task/query`, this endpoint returns only scheduling-relevant fields, excluding `task_input`, `metadata`, `result`, etc.

**Error Response:**

**400 Bad Request** - Invalid status filter:
```json
{
  "detail": {
    "success": false,
    "error": "Invalid status: invalid_status"
  }
}
```

**Example Usage:**

**curl:**
```bash
# Get all tasks
curl http://localhost:8000/task/schedule_info

# Filter by instance
curl "http://localhost:8000/task/schedule_info?instance_id=gpu-instance-1"

# Filter by status and model
curl "http://localhost:8000/task/schedule_info?status=pending&model_id=easyocr/dec"

# With pagination
curl "http://localhost:8000/task/schedule_info?limit=10&offset=20"
```

**Python:**
```python
import requests

scheduler_url = "http://localhost:8000"

# Get all schedule info
response = requests.get(f"{scheduler_url}/task/schedule_info")
data = response.json()
print(f"Total tasks: {data['total']}")

# Check instance distribution
instance_counts = {}
for task in data['tasks']:
    inst = task['assigned_instance'] or 'unassigned'
    instance_counts[inst] = instance_counts.get(inst, 0) + 1
print(f"Distribution: {instance_counts}")

# Get pending tasks for specific instance
response = requests.get(
    f"{scheduler_url}/task/schedule_info",
    params={"instance_id": "gpu-instance-1", "status": "pending"}
)
pending = response.json()
print(f"Pending on gpu-instance-1: {pending['count']}")
```

**Use Cases:**

1. **Instance Monitoring**: See how many tasks are assigned to each instance.

2. **Load Balancing Verification**: Verify that tasks are being distributed evenly across instances.

3. **Debugging**: Identify tasks stuck on a specific instance or find tasks that haven't been scheduled.

4. **Dashboard Integration**: Build monitoring dashboards showing real-time task distribution.

### POST `/callback/task_result`

**Internal API endpoint** for instances to report task completion results.

**Description:**

This endpoint is called by compute instances when a task completes or fails. It is **not intended for direct user access** - instances automatically invoke this endpoint as part of the task execution lifecycle.

**How it works:**
1. Scheduler dispatches task to instance via HTTP POST to instance's endpoint
2. Instance executes the task
3. Instance calls back to scheduler at `/callback/task_result` with the result
4. Scheduler updates task status and notifies WebSocket subscribers

**Request Body:**

**Completed Task:**
```json
{
  "task_id": "task-ocr-001",
  "status": "completed",
  "result": {
    "text": "Hello World",
    "confidence": 0.95,
    "bounding_boxes": [[10, 20, 100, 50]]
  },
  "error": null,
  "execution_time_ms": 3111
}
```

**Failed Task:**
```json
{
  "task_id": "task-ocr-002",
  "status": "failed",
  "result": null,
  "error": "Invalid image format: expected JPEG or PNG",
  "execution_time_ms": 223
}
```

**Field Descriptions:**
- `task_id` (required, string): ID of the completed task
- `status` (required, enum): Task completion status
  - `"completed"` - Task finished successfully with result
  - `"failed"` - Task encountered an error
- `result` (optional, object): Task output data
  - Required if status is `"completed"`
  - Should be `null` if status is `"failed"`
  - Structure is model-specific
- `error` (optional, string): Error message
  - Required if status is `"failed"`
  - Should be `null` if status is `"completed"`
- `execution_time_ms` (required, number): Actual execution time in milliseconds
  - Measured from when instance started processing until completion
  - Used for performance metrics and training data

**Success Response (200 OK):**
```json
{
  "success": true,
  "message": "Task result received successfully"
}
```

**Error Responses:**

**404 Not Found** - Task doesn't exist:
```json
{
  "success": false,
  "error": "Task not found"
}
```

**400 Bad Request** - Invalid status:
```json
{
  "success": false,
  "error": "Invalid status: pending. Must be 'completed' or 'failed'"
}
```

**Example:**
```bash
# This is typically called by the instance, not manually
curl -X POST http://localhost:8000/callback/task_result \
  -H "Content-Type: application/json" \
  -d '{
    "task_id": "task-ocr-001",
    "status": "completed",
    "result": {
      "text": "Hello World",
      "confidence": 0.95
    },
    "error": null,
    "execution_time_ms": 3111
  }'
```

**What Happens After Callback:**

1. **Task Registry Update**:
   - Task status changed to `completed` or `failed`
   - Result or error stored in task record
   - Completion timestamp recorded

2. **Instance Registry Update**:
   - Pending task count decremented
   - Completed or failed task count incremented
   - Queue info updated (removes completed task from queue)

3. **Training Data Collection** (if auto-training enabled):
   - Task metadata, execution time, and platform info sent to predictor service
   - Used to improve future runtime predictions

4. **WebSocket Notification**:
   - All clients subscribed to this task_id receive a result message
   - Message includes full task details (result, timestamps, execution time)

**Implementation Notes for Instance Developers:**

If you're implementing a compute instance, you should:

1. **Receive tasks** from the scheduler at your registered endpoint
2. **Execute the task** and measure execution time
3. **Call this callback** with the result or error
4. **Handle callback failures** with retry logic (3 retries recommended)

**Example Python Instance Implementation:**
```python
import httpx
import time

async def execute_and_report_task(scheduler_callback_url: str, task_data: dict):
    """Execute task and report result to scheduler."""
    task_id = task_data["task_id"]
    task_input = task_data["task_input"]

    # Execute task
    start_time = time.time()
    try:
        result = await execute_task(task_input)
        execution_time_ms = (time.time() - start_time) * 1000

        # Report success
        callback_data = {
            "task_id": task_id,
            "status": "completed",
            "result": result,
            "error": None,
            "execution_time_ms": execution_time_ms
        }
    except Exception as e:
        execution_time_ms = (time.time() - start_time) * 1000

        # Report failure
        callback_data = {
            "task_id": task_id,
            "status": "failed",
            "result": None,
            "error": str(e),
            "execution_time_ms": execution_time_ms
        }

    # Send callback to scheduler (with retries)
    async with httpx.AsyncClient() as client:
        for attempt in range(3):
            try:
                response = await client.post(
                    scheduler_callback_url,
                    json=callback_data,
                    timeout=10.0
                )
                response.raise_for_status()
                break
            except Exception as e:
                if attempt == 2:  # Last attempt
                    print(f"Failed to report task result after 3 attempts: {e}")
                await asyncio.sleep(1 * (attempt + 1))  # Exponential backoff
```

**Security Considerations:**

- This endpoint is currently **unauthenticated**
- In production, consider:
  - **API key authentication** to verify callbacks are from registered instances
  - **Rate limiting** to prevent abuse
  - **Request signing** to verify request integrity
  - **Network restrictions** (only accept callbacks from known instance IPs)

## Common Use Cases

### Submit Task and Poll for Result

```bash
# 1. Submit task
RESPONSE=$(curl -X POST http://localhost:8000/task/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_id": "task-001",
    "model_id": "easyocr/dec",
    "task_input": {"image": "..."},
    "metadata": {"height": 512, "width": 512}
  }')

# 2. Poll for completion (not recommended, use WebSocket instead)
while true; do
  TASK_INFO=$(curl "http://localhost:8000/task/info?task_id=task-001")
  STATUS=$(echo $TASK_INFO | jq -r '.task.status')

  if [ "$STATUS" = "completed" ] || [ "$STATUS" = "failed" ]; then
    echo "Task finished: $STATUS"
    echo $TASK_INFO | jq '.task.result'
    break
  fi

  sleep 0.5
done
```

**Note**: For real-time results, use [WebSocket API](7.WEBSOCKET_API.md) instead of polling.

## Performance Considerations

### Task Submission Latency

Task submission includes:
1. Validation (< 1ms)
2. Instance lookup (< 1ms)
3. **Predictor API call** (~10-100ms depending on network and predictor load)
4. Strategy selection (< 1ms)
5. Queue update (< 1ms)

**Optimization**: Enable predictor caching to reduce latency for repeated metadata patterns.

### Result Retrieval Methods

| Method | Latency | Efficiency | Use Case |
|--------|---------|------------|----------|
| Polling `/task/info` | 100-500ms avg | Low (wastes requests) | Simple scripts, rare tasks |
| WebSocket subscription | < 10ms | High (push-based) | Production, high throughput |

**Recommendation**: Always use WebSocket for production workloads.
