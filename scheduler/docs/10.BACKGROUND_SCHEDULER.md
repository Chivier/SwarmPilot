# 后台任务调度器

## 概述

BackgroundScheduler 是一个非阻塞任务调度系统，用于在后台异步处理CPU密集型的调度操作，确保API端点能够立即响应，防止事件循环阻塞和WebSocket超时。

**问题背景：**
在高负载场景（如500个workflows包含3500个tasks）下，同步调度会导致：
- API响应缓慢（20-70ms per task）
- 事件循环饥饿（无法处理WebSocket ping）
- WebSocket连接超时（1011错误）

**解决方案：**
将CPU密集操作移至后台异步执行，API立即返回，事件循环保持响应。

```
任务状态转换：

API调用阶段：
  submit → PENDING (assigned_instance="")
           ↓
           API返回 (5ms)

后台调度阶段：
  PENDING → (调度中...)
           ↓
           assigned_instance="inst-X"
           ↓
  RUNNING (任务已分发)
           ↓
  COMPLETED / FAILED
```

## 关键特性

### 1. 并发控制

使用 `asyncio.Semaphore` 限制并发调度数量：

```python
self._semaphore = asyncio.Semaphore(max_concurrent_scheduling)

async def _schedule_task_async(self, ...):
    async with self._semaphore:
        # 调度逻辑
        # 最多同时执行 max_concurrent_scheduling 个
```

**配置建议：**
- 默认值：50
- 低配置服务器：20-30
- 高配置服务器：100+

### 2. Fire-and-Forget 模式

```python
def schedule_task_background(self, task_id, ...):
    # 创建后台任务（不等待完成）
    task = asyncio.create_task(
        self._schedule_task_async(task_id, ...)
    )

    # 跟踪任务
    self._active_tasks[task_id] = task

    # 自动清理
    task.add_done_callback(
        lambda t: self._active_tasks.pop(task_id, None)
    )
```

### 3. 错误处理

所有后台调度错误都会：
1. 捕获异常
2. 将任务标记为 FAILED
3. 记录错误信息到 `task.error`
4. 记录详细日志

```python
try:
    # 调度逻辑
    ...
except Exception as e:
    await self.task_registry.update_status(task_id, TaskStatus.FAILED)
    await self.task_registry.set_error(task_id, f"Scheduling failed: {e}")
    logger.error(f"Task {task_id}: Scheduling failed - {e}")
```

### 4. 优雅关闭

```python
async def shutdown(self):
    """等待所有活动调度任务完成"""
    if self._active_tasks:
        logger.info(f"Waiting for {len(self._active_tasks)} tasks...")
        await asyncio.gather(
            *self._active_tasks.values(),
            return_exceptions=True
        )
```

## API 变更

### POST /task/submit

**修改前（同步调度）：**
```python
@app.post("/task/submit")
async def submit_task(request):
    # 1. 验证
    # 2. 获取实例
    available_instances = await instance_registry.list_active(...)

    # 3. 调度（阻塞 20-70ms）
    schedule_result = await scheduling_strategy.schedule_task(...)

    # 4. 创建任务（已分配实例）
    task = await task_registry.create_task(
        ...,
        assigned_instance=schedule_result.selected_instance_id
    )

    # 5. 更新统计
    await instance_registry.increment_pending(...)

    # 6. 分发任务
    task_dispatcher.dispatch_task_async(task_id)

    return {"task": task}  # 20-70ms 后返回
```

**修改后（后台调度）：**
```python
@app.post("/task/submit")
async def submit_task(request):
    # 1. 验证
    # 2. 快速检查实例存在
    available_instances = await instance_registry.list_active(...)
    if not available_instances:
        raise HTTPException(404)

    # 3. 创建任务（未分配实例）
    task = await task_registry.create_task(
        ...,
        assigned_instance="",  # 后台填充
    )

    # 4. 提交后台调度（非阻塞）
    background_scheduler.schedule_task_background(...)

    return {"task": task}  # ~5ms 返回
```

**关键变化：**
1. ✅ API响应时间：20-70ms → 5ms
2. ✅ 任务初始状态：assigned_instance="" （后台填充）
3. ✅ 事件循环不阻塞：可以立即处理WebSocket
4. ⚠️ 状态延迟：任务需要等待后台调度完成才分配实例

## 配置参数

### max_concurrent_scheduling

**默认值：** 50

**说明：** 同时执行的最大调度操作数

**调优建议：**

```python
# 低负载场景（< 100 tasks/min）
max_concurrent_scheduling = 20

# 中等负载（100-500 tasks/min）
max_concurrent_scheduling = 50  # 默认

# 高负载场景（> 500 tasks/min）
max_concurrent_scheduling = 100

# 计算公式：
# max_concurrent = (预期QPS × 平均调度时间) × 2
# 例如：(100 QPS × 0.05s) × 2 = 10
```

**影响因素：**
- CPU核心数
- predictor服务性能
- 网络延迟
- 内存限制

## 权衡和限制

### 优势

✅ **性能提升**
- API响应时间提升 4-14倍
- 总处理时间提升 2.5倍
- 事件循环保持响应

✅ **稳定性**
- 无WebSocket超时
- 错误隔离（单个任务失败不影响其他）
- 优雅关闭

✅ **可扩展性**
- 并发控制防止过载
- 适应高负载场景

### 权衡

⚠️ **状态延迟**
- 任务初始返回时 `assigned_instance=""`
- 需要等待后台调度完成（通常 <100ms）
- 客户端需要轮询或使用WebSocket获取最终状态

⚠️ **调试复杂度**
- 异步执行难以追踪
- 错误在后台发生，需要查看日志

⚠️ **资源使用**
- 每个后台任务消耗一个 asyncio Task
- 需要监控并发数避免过载

### 限制

❌ **不适用场景**
- 需要同步返回分配结果的场景
- 实时性要求极高（<10ms）的场景

❌ **依赖要求**
- Python 3.7+ (asyncio.create_task)
- FastAPI 异步支持

## 使用示例

### 客户端轮询

```python
import asyncio
import httpx

async def submit_and_wait(task_id, model_id, task_input):
    async with httpx.AsyncClient() as client:
        # 1. 提交任务（立即返回）
        resp = await client.post("/task/submit", json={
            "task_id": task_id,
            "model_id": model_id,
            "task_input": task_input,
            "metadata": {}
        })

        print(f"Task submitted: {resp.json()}")

        # 2. 轮询直到分配实例
        while True:
            await asyncio.sleep(0.1)  # 100ms

            task_resp = await client.get(f"/task/query?task_id={task_id}")
            task = task_resp.json()["task"]

            if task["assigned_instance"]:
                print(f"Assigned to: {task['assigned_instance']}")
                break
            elif task["status"] == "failed":
                print(f"Scheduling failed: {task['error']}")
                break
```

### 使用WebSocket（推荐）

```python
import websockets
import json

async def submit_with_websocket(task_id, model_id, task_input):
    async with websockets.connect("ws://localhost:8000/ws") as ws:
        # 1. 订阅任务
        await ws.send(json.dumps({
            "type": "subscribe",
            "task_ids": [task_id]
        }))

        # 2. 提交任务（立即返回）
        async with httpx.AsyncClient() as client:
            await client.post("/task/submit", json={
                "task_id": task_id,
                "model_id": model_id,
                "task_input": task_input,
                "metadata": {}
            })

        # 3. 等待结果（无需轮询）
        async for message in ws:
            msg = json.loads(message)
            if msg["type"] == "result":
                print(f"Task completed: {msg}")
                break
```

## 故障排查

### 问题1：任务长时间无assigned_instance

**症状：**
```python
task.assigned_instance == ""  # 持续很久
```

**可能原因：**
1. 后台调度队列满了（达到max_concurrent_scheduling）
2. predictor服务响应慢
3. 所有实例都在draining状态

**排查：**
```python
# 检查后台调度器状态
stats = await background_scheduler.get_stats()
print(stats["available_slots"])  # 应该 > 0

# 检查实例可用性
instances = await instance_registry.list_active(model_id)
print(f"Available instances: {len(instances)}")

# 检查日志
# grep "Scheduling failed" scheduler.log
```

### 问题2：任务被标记为FAILED

**症状：**
```python
task.status == "FAILED"
task.error  # "Scheduling failed: ..."
```

**常见原因：**
- No available instances
- Predictor service unavailable
- Invalid metadata

**解决：**
```bash
# 1. 检查实例状态
curl http://localhost:8000/instance/list

# 2. 检查predictor服务
curl http://localhost:8001/health

# 3. 查看详细日志
tail -f scheduler.log | grep ERROR
```
