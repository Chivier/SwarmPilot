# WebSocket API

Real-time, push-based task result notifications using WebSocket protocol.

> **See Also**:
> - [Data Models](2.DATA_MODELS.md) for message model definitions
> - [Task Management API](4.TASK_API.md) for task submission

---

## Overview

The WebSocket API provides low-latency, event-driven task result delivery. Instead of polling `/task/info`, clients subscribe to task IDs and receive results pushed immediately when tasks complete or fail.

**Benefits over polling:**
- **Lower latency**: Results delivered within milliseconds of completion
- **Reduced server load**: No repeated HTTP requests
- **Multiple subscriptions**: Monitor many tasks over a single connection
- **Immediate delivery**: Completed tasks sent instantly upon subscription

---

## Connection

### Endpoint

```
ws://scheduler-host:port/task/get_result
```

### Connection Flow

1. Client connects to WebSocket endpoint
2. Client sends subscribe message with task IDs
3. Server sends acknowledgment
4. Server pushes result messages when tasks complete/fail
5. Client optionally unsubscribes or closes connection

---

## Message Types

### Client → Server

#### Subscribe Message

Subscribe to task result updates.

```json
{
  "type": "subscribe",
  "task_ids": ["task-001", "task-002", "task-003"]
}
```

**Fields:**
- `type`: Must be `"subscribe"`
- `task_ids`: Array of task IDs to monitor (1-1000 tasks recommended per message)

**Behavior:**
- Tasks can be subscribed before or after submission
- If task is already completed, result is sent immediately
- Subscribing to non-existent tasks does not error (result sent when task exists)
- Multiple subscribe messages accumulate subscriptions

---

#### Unsubscribe Message

Unsubscribe from task result updates.

```json
{
  "type": "unsubscribe",
  "task_ids": ["task-001"]
}
```

**Fields:**
- `type`: Must be `"unsubscribe"`
- `task_ids`: Array of task IDs to stop monitoring

**Behavior:**
- Removes specified tasks from subscription list
- Does not affect other subscribed tasks
- Safe to unsubscribe from non-subscribed tasks (no-op)

---

#### Ping Message (Keepalive)

Application-level ping for connection keepalive.

```json
{
  "type": "ping",
  "timestamp": 1698765432.123
}
```

**Fields:**
- `type`: Always `"ping"`
- `timestamp`: Optional timestamp (seconds since epoch)

**When sent:**
- Server sends ping every 10 seconds to keep connection alive
- Client can send ping independently

**Client response:**
- Client should respond with pong message (optional but recommended)

---

#### Pong Message (Keepalive Response)

Application-level pong in response to ping.

```json
{
  "type": "pong",
  "timestamp": 1698765432.456
}
```

**Fields:**
- `type`: Always `"pong"`
- `timestamp`: Optional timestamp (seconds since epoch)

**When sent:**
- In response to server or client ping
- Can be sent independently as keepalive

---

### Server → Client

#### Acknowledgment Message

Confirms subscribe/unsubscribe operation.

```json
{
  "type": "ack",
  "message": "Subscribed to 3 tasks",
  "subscribed_tasks": ["task-001", "task-002", "task-003"]
}
```

**Fields:**
- `type`: Always `"ack"`
- `message`: Human-readable confirmation
- `subscribed_tasks`: Current list of all subscribed task IDs

**When sent:**
- After every subscribe message
- After every unsubscribe message

---

#### Task Result Message

Task completion or failure notification.

**Example - Completed Task:**
```json
{
  "type": "result",
  "task_id": "task-001",
  "status": "completed",
  "result": {
    "text": "Hello World",
    "confidence": 0.95
  },
  "error": null,
  "timestamps": {
    "submitted_at": "2025-10-31T12:34:56.789Z",
    "started_at": "2025-10-31T12:34:58.123Z",
    "completed_at": "2025-10-31T12:35:01.234Z"
  },
  "execution_time_ms": 3111
}
```

**Example - Failed Task:**
```json
{
  "type": "result",
  "task_id": "task-002",
  "status": "failed",
  "result": null,
  "error": "Invalid image format: expected JPEG or PNG",
  "timestamps": {
    "submitted_at": "2025-10-31T12:35:10.456Z",
    "started_at": "2025-10-31T12:35:12.789Z",
    "completed_at": "2025-10-31T12:35:13.012Z"
  },
  "execution_time_ms": 223
}
```

**Fields:**
- `type`: Always `"result"`
- `task_id`: ID of the completed task
- `status`: `"completed"` or `"failed"`
- `result`: Task output (null if failed)
- `error`: Error message (null if completed)
- `timestamps`: Task lifecycle timestamps
- `execution_time_ms`: Actual execution time

**When sent:**
- Immediately after subscription if task is already completed
- When a subscribed task completes or fails

---

#### Error Message

WebSocket or request error notification.

```json
{
  "type": "error",
  "error": "task_ids must be a list of strings",
  "task_id": null
}
```

**Fields:**
- `type`: Always `"error"`
- `error`: Error description
- `task_id`: Task ID if error is task-specific, otherwise null

**Common errors:**
- `"task_ids must be a list of strings"` - Invalid subscribe/unsubscribe format
- `"Unknown message type: foo"` - Invalid message type
- `"Server error: <details>"` - Internal server error

## Usage Examples

### Python (websockets library)

```python
import asyncio
import websockets
import json

async def monitor_tasks(task_ids):
    uri = "ws://localhost:8000/task/get_result"

    async with websockets.connect(uri) as websocket:
        # Subscribe to tasks
        subscribe_msg = {
            "type": "subscribe",
            "task_ids": task_ids
        }
        await websocket.send(json.dumps(subscribe_msg))

        # Receive acknowledgment
        ack = await websocket.recv()
        print(f"Subscription confirmed: {ack}")

        # Listen for results
        results = {}
        while len(results) < len(task_ids):
            message = await websocket.recv()
            data = json.loads(message)

            if data["type"] == "result":
                task_id = data["task_id"]
                status = data["status"]

                print(f"Task {task_id}: {status}")

                if status == "completed":
                    print(f"  Result: {data['result']}")
                    print(f"  Execution time: {data['execution_time_ms']}ms")
                else:
                    print(f"  Error: {data['error']}")

                results[task_id] = data

            elif data["type"] == "error":
                print(f"Error: {data['error']}")

        return results

# Usage
tasks = ["task-001", "task-002", "task-003"]
results = asyncio.run(monitor_tasks(tasks))
```

### JavaScript/TypeScript (Browser)

```javascript
class TaskMonitor {
  constructor(schedulerUrl) {
    this.ws = new WebSocket(`ws://${schedulerUrl}/task/get_result`);
    this.resultHandlers = new Map();
    this.setupListeners();
  }

  setupListeners() {
    this.ws.onmessage = (event) => {
      const data = JSON.parse(event.data);

      switch (data.type) {
        case 'ack':
          console.log('Subscribed to:', data.subscribed_tasks);
          break;

        case 'result':
          const handler = this.resultHandlers.get(data.task_id);
          if (handler) {
            handler(data);
            this.resultHandlers.delete(data.task_id);
          }
          break;

        case 'error':
          console.error('WebSocket error:', data.error);
          break;
      }
    };

    this.ws.onerror = (error) => {
      console.error('WebSocket connection error:', error);
    };

    this.ws.onclose = () => {
      console.log('WebSocket connection closed');
    };
  }

  subscribe(taskIds, onResult) {
    // Register result handlers
    taskIds.forEach(taskId => {
      this.resultHandlers.set(taskId, onResult);
    });

    // Send subscribe message
    this.ws.send(JSON.stringify({
      type: 'subscribe',
      task_ids: taskIds
    }));
  }

  unsubscribe(taskIds) {
    this.ws.send(JSON.stringify({
      type: 'unsubscribe',
      task_ids: taskIds
    }));

    taskIds.forEach(taskId => {
      this.resultHandlers.delete(taskId);
    });
  }

  close() {
    this.ws.close();
  }
}

// Usage
const monitor = new TaskMonitor('localhost:8000');

monitor.subscribe(['task-001', 'task-002'], (result) => {
  console.log(`Task ${result.task_id}: ${result.status}`);
  if (result.status === 'completed') {
    console.log('Result:', result.result);
  } else {
    console.error('Error:', result.error);
  }
});
```

### Node.js (ws library)

```javascript
const WebSocket = require('ws');

function monitorTasks(taskIds, onResult) {
  const ws = new WebSocket('ws://localhost:8000/task/get_result');

  ws.on('open', () => {
    // Subscribe to tasks
    ws.send(JSON.stringify({
      type: 'subscribe',
      task_ids: taskIds
    }));
  });

  ws.on('message', (data) => {
    const message = JSON.parse(data);

    if (message.type === 'ack') {
      console.log('Subscribed to:', message.subscribed_tasks);
    } else if (message.type === 'result') {
      onResult(message);
    } else if (message.type === 'error') {
      console.error('Error:', message.error);
    }
  });

  ws.on('error', (error) => {
    console.error('WebSocket error:', error);
  });

  ws.on('close', () => {
    console.log('WebSocket closed');
  });

  return ws;
}

// Usage
const ws = monitorTasks(['task-001', 'task-002'], (result) => {
  console.log(`Task ${result.task_id}: ${result.status}`);
  if (result.status === 'completed') {
    console.log('Result:', result.result);
    console.log(`Execution time: ${result.execution_time_ms}ms`);
  } else {
    console.error('Error:', result.error);
  }
});
```

## Subscription Patterns

### Pattern 1: Pre-calculated Task IDs (Recommended)

**Use Case**: When task IDs are known before task submission (most common scenario).

**Workflow:**
```
1. Pre-generate all task IDs
2. Subscribe to all task IDs via WebSocket
3. Wait for subscription acknowledgment
4. Submit tasks with the pre-generated IDs
5. Receive results as tasks complete
```

**Example:**
```python
import asyncio
import websockets
import requests
import json

async def run_workflow_with_precalculated_ids():
    # 1. Pre-generate task IDs
    task_ids = [f"workflow-{i:04d}-task" for i in range(100)]

    # 2. Connect and subscribe to ALL task IDs upfront
    uri = "ws://localhost:8000/task/get_result"
    async with websockets.connect(uri) as websocket:
        subscribe_msg = {
            "type": "subscribe",
            "task_ids": task_ids
        }
        await websocket.send(json.dumps(subscribe_msg))

        # Wait for acknowledgment
        ack = await websocket.recv()
        print(f"Subscribed to {len(task_ids)} tasks")

        # 3. Submit tasks using pre-generated IDs
        for task_id in task_ids:
            response = requests.post(
                "http://localhost:8000/task/submit",
                json={
                    "task_id": task_id,
                    "model_id": "my_model",
                    "task_input": {"data": "..."},
                    "metadata": {}
                }
            )

        # 4. Receive results as they complete
        results = {}
        while len(results) < len(task_ids):
            message = await websocket.recv()
            data = json.loads(message)

            if data["type"] == "result":
                results[data["task_id"]] = data
                print(f"Received result {len(results)}/{len(task_ids)}")

        return results

# Run
results = asyncio.run(run_workflow_with_precalculated_ids())
```

**Advantages:**
- ✅ Zero subscription overhead during execution
- ✅ Guaranteed to receive all results (no race conditions)
- ✅ Simple and efficient (single subscription message)
- ✅ Works perfectly with dynamic task generation patterns

**Performance:**
- Subscription overhead: ~1ms (single message for all tasks)
- No per-task subscription cost
- Ideal for batch workloads and workflows

---

### Pattern 2: Incremental Subscription

**Use Case**: When task IDs are generated dynamically and cannot be pre-calculated.

**Workflow:**
```
1. Connect to WebSocket
2. For each new task:
   a. Generate task ID
   b. Subscribe to task ID
   c. Submit task
3. Receive results as tasks complete
```

**Advantages:**
- ✅ Supports truly dynamic task ID generation
- ✅ No need to pre-calculate task IDs

**Disadvantages:**
- ⚠️ Higher overhead: ~0.5-1ms per task for subscription
- ⚠️ 100 tasks = ~50-100ms total subscription overhead

---

### Pattern 3: Batch Subscription

**Use Case**: Balance between pre-calculation and full dynamic generation.

**Workflow:**
```
1. Connect to WebSocket
2. Generate tasks in batches (e.g., 10-50 at a time)
3. Subscribe to batch
4. Submit batch
5. Repeat for next batch
```

**Performance:**
```
100 tasks, batch_size=20:
  - Subscription messages: 5 (vs 1 for pre-calculated, 100 for incremental)
  - Overhead: ~5ms (vs ~1ms for pre-calculated, ~80ms for incremental)
```

## Best Practices

### Subscribe Before Submitting Tasks

To avoid race conditions, subscribe before submitting tasks:

```python
# 1. Connect and subscribe
subscribe_msg = {"type": "subscribe", "task_ids": ["task-001"]}
await websocket.send(json.dumps(subscribe_msg))
await websocket.recv()  # Wait for ack

# 2. Submit task via HTTP
response = await http_client.post("/task/submit", json={...})

# 3. Receive result via WebSocket
result = await websocket.recv()
```

### Batch Subscriptions

Subscribe to multiple tasks in a single message for efficiency:

```python
# Good: Single message
await websocket.send(json.dumps({
    "type": "subscribe",
    "task_ids": ["task-001", "task-002", "task-003"]
}))

# Avoid: Multiple messages (slower)
for task_id in ["task-001", "task-002", "task-003"]:
    await websocket.send(json.dumps({
        "type": "subscribe",
        "task_ids": [task_id]
    }))
```

### Clean Up Subscriptions

Unsubscribe from tasks you no longer need to reduce memory usage:

```python
# Process result and unsubscribe
if data["type"] == "result":
    process_result(data)

    # Unsubscribe from completed task
    await websocket.send(json.dumps({
        "type": "unsubscribe",
        "task_ids": [data["task_id"]]
    }))
```

## Error Handling

### Connection Errors

Handle connection failures gracefully:

```python
try:
    async with websockets.connect(uri) as websocket:
        # ... use websocket
except websockets.exceptions.ConnectionClosed:
    print("Connection closed unexpectedly")
    # Implement reconnection logic
except Exception as e:
    print(f"WebSocket error: {e}")
```

### Reconnection Strategy

Implement reconnection logic with exponential backoff:

```python
import asyncio
import websockets

async def connect_with_retry(uri, max_retries=5):
    retry_delay = 1

    for attempt in range(max_retries):
        try:
            websocket = await websockets.connect(uri)
            print("Connected to scheduler")
            return websocket
        except Exception as e:
            print(f"Connection failed (attempt {attempt + 1}/{max_retries}): {e}")
            if attempt < max_retries - 1:
                await asyncio.sleep(retry_delay)
                retry_delay *= 2  # Exponential backoff

    raise Exception("Failed to connect after maximum retries")
```

## Performance Considerations

### Latency

Typical result delivery latency:
- **Local network**: < 5ms from task completion to client receipt
- **Internet**: 10-50ms depending on network conditions
- **Polling alternative**: 100-500ms average (depends on poll interval)

### Scalability

- Each WebSocket connection consumes ~10KB memory
- Server can handle thousands of concurrent connections
- Subscribe to 100-1000 tasks per connection recommended
- For massive scale (10,000+ tasks), use multiple connections

### Message Ordering

Results are delivered in the order tasks complete, not submission order. If order matters, track timestamps in client.
