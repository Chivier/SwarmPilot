# Health Check API

This document describes the health check endpoint for monitoring scheduler service status.

> **See Also**: [Data Models](2.DATA_MODELS.md) for response model definitions

---

## Overview

The health check endpoint provides a simple way to verify that the scheduler service is running and to monitor its current state. This endpoint is commonly used for:

- **Load balancer health checks**: Verify service is accepting requests
- **Monitoring systems**: Track service uptime and status
- **Operational dashboards**: Display real-time service statistics
- **Deployment validation**: Confirm successful service startup

---

## Endpoint

### GET `/health`

Check the health status of the scheduler service.

**Query Parameters:** None

**Example Request:**
```
GET /health
```

**Success Response (200 OK):**
```json
{
  "success": true,
  "status": "healthy",
  "timestamp": "2025-10-31T12:34:56.789Z",
  "version": "1.0.0",
  "stats": {
    "total_instances": 5,
    "active_instances": 5,
    "total_tasks": 1523,
    "pending_tasks": 12,
    "running_tasks": 8,
    "completed_tasks": 1485,
    "failed_tasks": 18
  }
}
```

**Response Fields:**

- `success` (boolean): Always `true` for successful health checks
- `status` (string): Service health status
  - `"healthy"` - Service is operational
  - `"unhealthy"` - Service has issues (returns 503)
- `timestamp` (string): Current server time in ISO 8601 format with timezone
- `version` (string): Scheduler service version
- `stats` (object): Real-time service statistics
  - `total_instances` (int): Total registered instances
  - `active_instances` (int): Instances currently available (same as total_instances)
  - `total_tasks` (int): Total tasks ever submitted
  - `pending_tasks` (int): Tasks assigned but not yet dispatched
  - `running_tasks` (int): Tasks currently executing on instances
  - `completed_tasks` (int): Successfully completed tasks
  - `failed_tasks` (int): Tasks that failed with errors

**Error Response (503 Service Unavailable):**
```json
{
  "success": false,
  "status": "unhealthy",
  "error": "Predictor service unavailable",
  "timestamp": "2025-10-31T12:34:56.789Z"
}
```

**Error Fields:**
- `success`: Always `false` for unhealthy state
- `status`: Always `"unhealthy"`
- `error`: Description of the health check failure
- `timestamp`: Current server time

**Example Usage:**
```bash
# Basic health check
curl "http://localhost:8000/health"

# Pretty-print with jq
curl "http://localhost:8000/health" | jq

# Extract specific stats
curl "http://localhost:8000/health" | jq '.stats.running_tasks'
```

## Statistics Interpretation

### Instance Metrics

**total_instances**: Number of registered instances
- Increases when instances are registered via `/instance/register`
- Decreases when instances are removed via `/instance/remove`

**active_instances**: Currently equal to `total_instances`
- Future enhancement: Track instance availability based on heartbeats

**Interpretation:**
- If `total_instances` is 0, no instances are available to execute tasks
- Task submission will fail with "No available instance" error

### Task Metrics

**total_tasks**: Cumulative count of all tasks ever submitted
- Monotonically increasing
- Resets to 0 when tasks are cleared via `/task/clear`

**pending_tasks**: Tasks assigned to instances but not yet dispatched
- Should be 0 or very low under normal operation
- High values indicate dispatching bottleneck

**running_tasks**: Tasks currently executing on instances
- Reflects current system load
- Maximum value depends on instance capacity

**completed_tasks**: Successfully finished tasks
- Increases as tasks complete

**failed_tasks**: Tasks that encountered errors
- Monitor this for quality issues

### Health Indicators

**Good Health:**
```json
{
  "pending_tasks": 2,
  "running_tasks": 8,
  "completed_tasks": 1485,
  "failed_tasks": 5
}
```
- Low pending count (tasks dispatched quickly)
- Moderate running count (system is working)
- High completed count (productive)
- Low failed count (<1% failure rate)

**Potential Issues:**
```json
{
  "pending_tasks": 500,
  "running_tasks": 2,
  "completed_tasks": 100,
  "failed_tasks": 50
}
```
- High pending count (dispatching is slow or instances are overloaded)
- Low running count (instances may be down or not accepting tasks)
- High failure rate (50/(100+50) = 33%)

**Action Items:**
- Check instance availability via `/instance/list`
- Review recent failures via `/task/list?status=failed`
- Monitor instance logs for errors
- Consider adding more instances

## Integration Examples

### Load Balancer Health Check

**HAProxy Example:**
```
backend scheduler_backend
    option httpchk GET /health
    http-check expect status 200
    server scheduler1 192.168.1.10:8000 check inter 5s
    server scheduler2 192.168.1.11:8000 check inter 5s
```

**NGINX Example:**
```nginx
upstream scheduler {
    server 192.168.1.10:8000 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8000 max_fails=3 fail_timeout=30s;

    # Health check (requires nginx-plus or custom module)
    health_check interval=5s uri=/health;
}
```

### Kubernetes Liveness and Readiness Probes

**Deployment YAML:**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: scheduler
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: scheduler
        image: scheduler:1.0.0
        ports:
        - containerPort: 8000

        # Readiness probe - service ready to accept traffic
        readinessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 5
          periodSeconds: 10
          timeoutSeconds: 2
          successThreshold: 1
          failureThreshold: 3

        # Liveness probe - service is still alive
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 15
          periodSeconds: 20
          timeoutSeconds: 5
          successThreshold: 1
          failureThreshold: 3
```

**Explanation:**
- **Readiness probe**: Checks every 10s, fails after 3 consecutive failures
  - Pod marked as "not ready" → removed from service load balancing
- **Liveness probe**: Checks every 20s, fails after 3 consecutive failures
  - Pod marked as "unhealthy" → Kubernetes restarts the container

### Monitoring Script

**Simple Shell Script:**
```bash
#!/bin/bash
# check_scheduler.sh - Monitor scheduler uptime

SCHEDULER_URL="http://localhost:8000/health"
LOG_FILE="/var/log/scheduler_health.log"

while true; do
  TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

  # Check health
  HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "$SCHEDULER_URL")

  if [ "$HTTP_CODE" -eq 200 ]; then
    # Get stats
    STATS=$(curl -s "$SCHEDULER_URL" | jq -c '.stats')
    echo "$TIMESTAMP OK $STATS" >> "$LOG_FILE"
  else
    echo "$TIMESTAMP FAIL HTTP_CODE=$HTTP_CODE" >> "$LOG_FILE"

    # Send alert (example: Slack webhook)
    curl -X POST https://hooks.slack.com/services/YOUR/WEBHOOK/URL \
      -H 'Content-Type: application/json' \
      -d "{\"text\": \"Scheduler health check failed: HTTP $HTTP_CODE\"}"
  fi

  sleep 60  # Check every minute
done
```

**Cron Job (Check every 5 minutes):**
```cron
*/5 * * * * /usr/local/bin/check_scheduler.sh
```

### Real-time Monitoring Dashboard

**Python Example:**
```python
#!/usr/bin/env python3
import time
import requests
from rich.console import Console
from rich.table import Table
from rich.live import Live

console = Console()

def create_stats_table(data):
    """Create a formatted stats table."""
    table = Table(title="Scheduler Health Stats", show_header=True)
    table.add_column("Metric", style="cyan")
    table.add_column("Value", style="green", justify="right")

    stats = data.get('stats', {})
    table.add_row("Status", data.get('status', 'unknown'))
    table.add_row("Version", data.get('version', 'unknown'))
    table.add_row("", "")  # Separator
    table.add_row("Total Instances", str(stats.get('total_instances', 0)))
    table.add_row("Active Instances", str(stats.get('active_instances', 0)))
    table.add_row("", "")  # Separator
    table.add_row("Total Tasks", str(stats.get('total_tasks', 0)))
    table.add_row("Pending", str(stats.get('pending_tasks', 0)))
    table.add_row("Running", str(stats.get('running_tasks', 0)))
    table.add_row("Completed", str(stats.get('completed_tasks', 0)))
    table.add_row("Failed", str(stats.get('failed_tasks', 0)))

    # Calculate success rate
    total = stats.get('completed_tasks', 0) + stats.get('failed_tasks', 0)
    if total > 0:
        success_rate = (stats.get('completed_tasks', 0) / total) * 100
        table.add_row("", "")  # Separator
        table.add_row("Success Rate", f"{success_rate:.2f}%")

    return table

def monitor():
    """Monitor scheduler health in real-time."""
    with Live(console=console, refresh_per_second=1) as live:
        while True:
            try:
                response = requests.get('http://localhost:8000/health', timeout=5)
                if response.status_code == 200:
                    data = response.json()
                    table = create_stats_table(data)
                    live.update(table)
                else:
                    live.update(f"[red]Health check failed: HTTP {response.status_code}[/red]")
            except Exception as e:
                live.update(f"[red]Error: {e}[/red]")

            time.sleep(2)

if __name__ == '__main__':
    monitor()
```

**Run the monitor:**
```bash
pip install rich requests
python3 monitor_scheduler.py
```

## Performance

The health check endpoint is designed to be fast:
- **Typical response time**: < 5ms
- **Operations performed**:
  - Count instances in registry (O(1))
  - Count tasks by status (O(1) with indexing)
  - Format response (< 1ms)

**Note**: If predictor health check is enabled (currently commented out), response time may increase by 10-100ms depending on network latency.

## Security Considerations

### Public Exposure

The health endpoint exposes operational metrics that could be useful to attackers:
- Instance count (infrastructure scale)
- Task throughput (usage patterns)
- Failure rates (potential vulnerabilities)

**Recommendations:**
1. **Internal network only**: Restrict health endpoint to internal monitoring systems
2. **Authentication**: Add API key or basic auth for external exposure
3. **Rate limiting**: Prevent abuse via excessive health checks

**Example: NGINX Rate Limiting**
```nginx
location /health {
    limit_req zone=health_limit burst=5 nodelay;
    proxy_pass http://scheduler;
}

# Define rate limit zone
limit_req_zone $binary_remote_addr zone=health_limit:10m rate=10r/m;
```

### Information Disclosure

Consider creating a separate **public health endpoint** with minimal information:

```python
@app.get("/health/public")
async def public_health():
    """Public health check with minimal information."""
    return {
        "status": "healthy",
        "timestamp": datetime.now().isoformat() + "Z"
    }
```

Use this for external load balancers and keep detailed stats internal.
