# Strategy Management API

This document describes the API endpoints for managing scheduling strategies.

> **See Also**:
> - [Data Models](2.DATA_MODELS.md) for request/response model definitions
> - [Scheduling Strategies](8.SCHEDULING_STRATEGIES.md) for algorithm details

---

## Overview

The scheduler supports multiple scheduling strategies that determine how tasks are assigned to instances. You can query the current strategy and switch between strategies at runtime.

**Available Strategies:**
- **round_robin** - Distribute tasks evenly across instances in rotation
- **min_time** - Select instance with shortest expected queue completion time
- **probabilistic** - Use probabilistic sampling based on queue time distributions

---

## Endpoints

### GET `/strategy/get`

Get the current scheduling strategy and its parameters.

**Query Parameters:** None

**Example Request:**
```
GET /strategy/get
```

**Success Response (200 OK):**

**Round Robin Strategy:**
```json
{
  "success": true,
  "strategy_info": {
    "strategy_name": "round_robin",
    "parameters": {}
  }
}
```

**Minimum Time Strategy:**
```json
{
  "success": true,
  "strategy_info": {
    "strategy_name": "min_time",
    "parameters": {}
  }
}
```

**Probabilistic Strategy:**
```json
{
  "success": true,
  "strategy_info": {
    "strategy_name": "probabilistic",
    "parameters": {
      "target_quantile": 0.9
    }
  }
}
```

**Response Fields:**
- `strategy_name`: Name of the current strategy (`"round_robin"`, `"min_time"`, or `"probabilistic"`)
- `parameters`: Strategy-specific configuration
  - `target_quantile` (probabilistic only): Target quantile for scheduling decisions (0.0-1.0)

**Example:**
```bash
curl "http://localhost:8000/strategy/get"
```

---

### POST `/strategy/set`

Set the scheduling strategy for the scheduler.

**Important**: This operation clears all tasks and reinitializes instance queues.

**Request Body:**

**Switch to Round Robin:**
```json
{
  "strategy_name": "round_robin"
}
```

**Switch to Minimum Time:**
```json
{
  "strategy_name": "min_time"
}
```

**Switch to Probabilistic (with custom quantile):**
```json
{
  "strategy_name": "probabilistic",
  "target_quantile": 0.95
}
```

**Field Descriptions:**
- `strategy_name` (required, enum): Strategy to switch to
  - `"round_robin"` - Round-robin distribution
  - `"min_time"` - Shortest expected queue time
  - `"probabilistic"` - Probabilistic sampling
- `target_quantile` (optional, float, default=0.9): Target quantile for probabilistic strategy (0.0-1.0)
  - `0.5` = optimize for median (50th percentile)
  - `0.9` = optimize for 90th percentile (default)
  - `0.99` = optimize for 99th percentile (tail latency)

**Success Response (200 OK):**
```json
{
  "success": true,
  "message": "Successfully switched to 'probabilistic' strategy",
  "cleared_tasks": 15,
  "reinitialized_instances": 3,
  "strategy_info": {
    "strategy_name": "probabilistic",
    "parameters": {
      "target_quantile": 0.9
    }
  }
}
```

**Response Fields:**
- `message`: Human-readable confirmation message
- `cleared_tasks`: Number of tasks removed during strategy switch
- `reinitialized_instances`: Number of instances whose queue info was reset
- `strategy_info`: New strategy configuration

**Error Responses:**

**400 Bad Request** - Tasks currently running:
```json
{
  "success": false,
  "error": "Cannot switch strategy while 5 task(s) are running. Please wait for tasks to complete or fail them first."
}
```

**500 Internal Server Error** - Strategy initialization failed:
```json
{
  "success": false,
  "error": "Failed to initialize strategy: Invalid parameter"
}
```

**Example Usage:**
```bash
# Switch to probabilistic with default quantile (0.9)
curl -X POST http://localhost:8000/strategy/set \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "probabilistic"}'

# Switch to probabilistic optimizing for 95th percentile
curl -X POST http://localhost:8000/strategy/set \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "probabilistic", "target_quantile": 0.95}'

# Switch to round robin
curl -X POST http://localhost:8000/strategy/set \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "round_robin"}'
```

## Strategy Switching Workflow

### What Happens During Strategy Switch

When you set a new strategy, the scheduler performs the following operations:

1. **Validate running tasks**: Checks if any tasks are currently running
   - If tasks are running, the request is rejected with 400 error
   - You must wait for tasks to complete or manually fail them

2. **Clear all tasks**: Removes all task records from the task registry
   - Pending tasks are discarded
   - Completed/failed task history is cleared

3. **Reinitialize instance queues**: Resets queue information for all instances
   - **min_time strategy**: Creates `InstanceQueueExpectError` with `expected_time_ms` and `error_margin_ms`
   - **probabilistic/round_robin**: Creates `InstanceQueueProbabilistic` with `quantiles` and `values`

4. **Update strategy**: Switches to the new scheduling strategy

5. **Update configuration**: Updates in-memory configuration (not persisted to disk)

### Safe Strategy Switch Procedure

```bash
# 1. Check current strategy
curl "http://localhost:8000/strategy/get"

# 2. Check for running tasks
curl "http://localhost:8000/task/list?status=running"

# 3. Wait for running tasks to complete, or clear all tasks
curl -X POST "http://localhost:8000/task/clear"

# 4. Switch strategy
curl -X POST "http://localhost:8000/strategy/set" \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "probabilistic", "target_quantile": 0.95}'

# 5. Verify strategy change
curl "http://localhost:8000/strategy/get"
```

## Strategy Selection Guide

### Round Robin

**When to use:**
- Instances have similar performance characteristics
- Simple load balancing without runtime prediction
- Development/testing environments

**Advantages:**
- Simple and predictable
- No dependency on predictor service
- Fair distribution across instances

**Disadvantages:**
- Ignores queue length and task characteristics
- May create bottlenecks on slower instances

**Queue Info Type:** `InstanceQueueProbabilistic` (not actively used for scheduling)

---

### Minimum Time (Shortest Queue)

**When to use:**
- Minimizing average task completion time
- Instances with varying performance
- Production workloads with performance requirements

**Advantages:**
- Optimizes for expected completion time
- Considers queue length and task predictions
- Adapts to instance performance differences

**Disadvantages:**
- May overload fast instances
- Requires accurate runtime predictions
- Sensitive to prediction errors

**Queue Info Type:** `InstanceQueueExpectError`
- `expected_time_ms`: Mean queue completion time
- `error_margin_ms`: Standard deviation

**Selection Logic:**
1. Calculate total queue time for each instance (queue + predicted task time)
2. Select instance with minimum expected time
3. If multiple instances have same expected time, choose minimum error margin

---

### Probabilistic

**When to use:**
- Optimizing for tail latency (p95, p99)
- Production workloads with strict SLAs
- Workloads with high variability

**Advantages:**
- Optimizes for specific latency percentiles
- Handles runtime variability well
- Balances load probabilistically

**Disadvantages:**
- More complex than other strategies
- Requires accurate quantile predictions
- Slightly higher computational cost

**Queue Info Type:** `InstanceQueueProbabilistic`
- `quantiles`: `[0.5, 0.9, 0.95, 0.99]`
- `values`: Estimated queue completion times at each quantile

**Selection Logic:**
1. For each instance, sample queue completion time from quantile distribution
2. Add predicted task time to sampled queue time
3. Select instance with minimum sampled total time
4. Sampling creates probabilistic load balancing

**Parameters:**
- `target_quantile` (default: 0.9):
  - `0.5`: Optimize for median latency
  - `0.9`: Optimize for 90th percentile (good balance)
  - `0.95`: Optimize for 95th percentile
  - `0.99`: Optimize for 99th percentile (tail latency)

## Performance Impact

### Strategy Switching Cost

- **Validation**: < 1ms
- **Task clearing**: ~0.1ms per task
- **Queue reinitialization**: ~0.5ms per instance
- **Strategy creation**: < 5ms

**Example**: Switching strategy with 100 tasks and 5 instances takes ~20ms.

### Runtime Performance

| Strategy | Selection Time | Predictor Calls | Complexity |
|----------|---------------|-----------------|------------|
| Round Robin | < 0.1ms | 0 | O(1) |
| Minimum Time | < 0.5ms | N (instances) | O(N) |
| Probabilistic | < 1ms | N (instances) | O(N) |

All strategies are suitable for production use.

## Common Use Cases

### Development: Start with Round Robin

```bash
# Simple load balancing for testing
curl -X POST http://localhost:8000/strategy/set \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "round_robin"}'
```

### Production: Optimize Average Latency

```bash
# Use minimum time for best average performance
curl -X POST http://localhost:8000/strategy/set \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "min_time"}'
```

### Production: Optimize Tail Latency (SLA)

```bash
# Use probabilistic for p95 SLA
curl -X POST http://localhost:8000/strategy/set \
  -H "Content-Type: application/json" \
  -d '{"strategy_name": "probabilistic", "target_quantile": 0.95}'
```

### Benchmarking: Compare Strategies

```bash
# Test each strategy with same workload
for strategy in round_robin min_time probabilistic; do
  echo "Testing $strategy..."

  # Switch strategy
  curl -X POST http://localhost:8000/strategy/set \
    -H "Content-Type: application/json" \
    -d "{\"strategy_name\": \"$strategy\"}"

  # Run benchmark
  python benchmark.py --workload standard

  # Collect metrics
  curl "http://localhost:8000/health" | jq .stats
done
```

## Important Notes

### Data Loss Warning

**Switching strategies clears all tasks:**
- Pending tasks are discarded (not executed)
- Running tasks are blocked from switching (returns 400 error)
- Completed task history is permanently deleted
- Instance registrations are preserved
- Instance statistics (completed/failed counts) are preserved

**Recommendation**: Only switch strategies during maintenance windows or when the task queue is empty.

### Configuration Persistence

Strategy changes are **not persisted** to configuration files:
- Changes only affect the current runtime session
- After scheduler restart, the default strategy from config is used
- To persist changes, manually update the configuration file:

```yaml
# config.yaml
scheduling:
  default_strategy: "probabilistic"
  probabilistic_quantile: 0.95
```

### Queue Info Compatibility

Each strategy requires specific queue info types:
- **min_time** → `InstanceQueueExpectError`
- **probabilistic/round_robin** → `InstanceQueueProbabilistic`

Switching strategies automatically reinitializes queue info to the correct type. Existing queue state is **discarded**.

## Error Handling

All endpoints follow a consistent error response format:

```json
{
  "success": false,
  "error": "Error description here"
}
```

**Common HTTP Status Codes:**
- **200 OK**: Request succeeded
- **400 Bad Request**: Invalid request (e.g., tasks are running)
- **500 Internal Server Error**: Strategy initialization failed
