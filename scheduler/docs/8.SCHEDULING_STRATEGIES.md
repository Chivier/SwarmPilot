# Scheduling Strategies

This document provides detailed explanations of the scheduling algorithms and their implementation.

> **See Also**:
> - [Strategy Management API](5.STRATEGY_API.md) for switching strategies
> - [Data Models](2.DATA_MODELS.md) for queue information models

---

## Overview

The scheduler supports three scheduling strategies that determine how tasks are assigned to instances:

1. **Round Robin** - Simple cyclic distribution
2. **Minimum Time (Shortest Queue)** - Greedy selection based on expected completion time
3. **Probabilistic** - Quantile-based selection for tail latency optimization

Each strategy uses different queue information models and selection algorithms.

---

## Round Robin Strategy

### Algorithm

Distributes tasks evenly across instances in rotation.

**Selection Logic:**
```
1. Track current position in instance list
2. Assign task to instance at current position
3. Move position to next instance (circular)
4. Repeat
```

### Queue Information

**Type**: `InstanceQueueProbabilistic`

**Note**: Queue information is maintained but **not used** for selection decisions. Round robin ignores queue state and runtime predictions.

### When to Use

**Best for:**
- Instances with similar performance characteristics
- Simple load balancing without runtime prediction
- Development/testing environments
- Fallback when predictor service is unavailable

**Advantages:**
- ✅ Simple and predictable
- ✅ No dependency on predictor service
- ✅ Fair distribution across instances
- ✅ O(1) selection time

**Disadvantages:**
- ❌ Ignores queue length and task characteristics
- ❌ May create bottlenecks on slower instances
- ❌ No optimization for latency or throughput

### Example

```
Instances: [A, B, C]
Tasks:     [T1, T2, T3, T4, T5, T6]

Assignment:
  T1 → A  (position = 0)
  T2 → B  (position = 1)
  T3 → C  (position = 2)
  T4 → A  (position = 0, wrapped around)
  T5 → B  (position = 1)
  T6 → C  (position = 2)
```

Even if instance A is much slower than B and C, tasks are still distributed equally.

## Minimum Time Strategy (Shortest Queue)

### Algorithm

Selects the instance with the shortest expected queue completion time.

**Selection Logic:**
```
1. For each instance:
   a. Get current queue expected time (Eq) and error margin (σq)
   b. Get predicted task time (Et) and error margin (σt)
   c. Calculate total expected time: E_total = Eq + σq + Et
2. Select instance with minimum E_total
3. If multiple instances have same E_total, choose first one
```

之前的实现存在bug，只考虑了队列时间而忽略了任务预测时间：
```python
# 错误实现（已修复）：
queue_metrics = min([[queue.expected_time_ms, queue.error_margin_ms, instance_id]
                     for instance_id, queue in queue_info.items()])
best_instance_id = queue_metrics[2]  # ❌ 忽略了任务预测时间
```

**正确实现（v2.0+）：**
```python

for pred in predictions:
    queue = queue_info.get(pred.instance_id)
    if queue:
        # ✅ 正确：队列时间 + 队列误差 + 任务预测时间
        total_time = queue.expected_time_ms + queue.error_margin_ms + pred.predicted_time_ms
    else:
        total_time = pred.predicted_time_ms

    if total_time < best_total_time:
        best_total_time = total_time
        best_instance_id = pred.instance_id
```

此bug会导致：
- ❌ 忽略任务预测时间，总是选择队列最短的实例
- ❌ 即使某个实例对该任务预测时间很长，也会被选中
- ❌ 调度决策不optimal

已在 commit `3d4944b` 中修复。

### Queue Information

**Type**: `InstanceQueueExpectError`

**Fields:**
- `expected_time_ms`: Expected queue completion time (mean)
- `error_margin_ms`: Uncertainty/error margin (standard deviation)

### Queue Update

**On Task Submission:**

Uses error propagation formula for independent random variables:

```
E_new = E_queue + E_task
σ_new = sqrt(σ_queue² + σ_task²)
```

**Implementation** (`src/api.py:433-458`):
```python
new_expected = current_queue.expected_time_ms + task_expected
new_error = math.sqrt(
    current_queue.error_margin_ms ** 2 + task_error ** 2
)
```

**On Task Completion:**

Adjusts expected time based on actual execution:

```
E_new = E_queue - E_predicted + E_actual
σ_new = σ_queue  (unchanged)
```

**Implementation** (`src/task_dispatcher.py:266-278`):
```python
new_expected = current_queue.expected_time_ms - predicted_time_ms + actual_time_ms
new_expected = max(0.0, new_expected)  # Ensure non-negative

updated_queue = InstanceQueueExpectError(
    instance_id=instance_id,
    expected_time_ms=new_expected,
    error_margin_ms=current_queue.error_margin_ms,  # Keep unchanged
)
```

**Rationale:**
- Expected time is adjusted by removing predicted contribution and adding actual time
- Error margin represents queue uncertainty and is not updated per individual task

### When to Use

**Best for:**
- Minimizing average task completion time
- Instances with varying performance
- Production workloads with performance requirements

**Advantages:**
- ✅ Optimizes for expected completion time
- ✅ Considers queue length and task predictions
- ✅ Adapts to instance performance differences
- ✅ Fast selection: O(N) where N = number of instances

**Disadvantages:**
- ❌ May overload fast instances
- ❌ Requires accurate runtime predictions
- ❌ Sensitive to prediction errors
- ❌ Optimizes for average, not tail latency

### Example

```
Instances with current queues:
  A: E=100ms, σ=10ms
  B: E=150ms, σ=5ms
  C: E=120ms, σ=15ms

New task prediction:
  E_task=50ms, σ_task=8ms

Total expected times:
  A: 100 + 50 = 150ms
  B: 150 + 50 = 200ms
  C: 120 + 50 = 170ms

Selected: Instance A (minimum total time: 150ms)

After assignment, A's queue:
  E_new = sqrt(10² + 8²) = sqrt(164) ≈ 12.8ms
  New queue: E=150ms, σ=12.8ms
```

## Probabilistic Strategy

### Algorithm

Selects instances based on probabilistic sampling from quantile distributions.

**Selection Logic:**
```
1. For each instance:
   a. Get queue quantile distribution (quantiles, values)
   b. Sample queue time from distribution
   c. Add predicted task time to sampled queue time
   d. Result: sampled_total_time
2. Select instance with minimum sampled_total_time
```

**Sampling creates probabilistic load balancing:**
- Fast instances with low queue times are selected more often
- Slow instances with high queue times are selected less often
- Natural load balancing emerges from sampling

### Queue Information

**Type**: `InstanceQueueProbabilistic`

**Fields:**
- `quantiles`: List of quantile positions `[0.5, 0.9, 0.95, 0.99]`
- `values`: Estimated queue completion times at each quantile (ms)

**Interpretation:**
```json
{
  "quantiles": [0.5, 0.9, 0.95, 0.99],
  "values": [120.5, 250.3, 300.8, 450.2]
}
```
- 50% chance queue completes in ≤120.5ms
- 90% chance queue completes in ≤250.3ms
- 95% chance queue completes in ≤300.8ms
- 99% chance queue completes in ≤450.2ms

### Queue Update

**Method**: Monte Carlo sampling with 1000 samples

**On Task Submission** (`src/api.py:460-513`):

```
1. Sample 1000 values from current queue distribution
2. Sample 1000 values from task prediction distribution
3. Add samples element-wise: total_samples = queue_samples + task_samples
4. Compute new quantiles from total_samples
```

**On Task Completion** (`src/task_dispatcher.py:280-335`):

```
1. Sample 1000 values from current queue distribution
2. Sample 1000 values from predicted task distribution
3. Compute updated queue: updated_samples = queue_samples - task_samples + actual_time_ms
4. Ensure all values are non-negative
5. Compute new quantiles from updated_samples
```

### Parameters

**target_quantile** (default: 0.9, range: 0.0-1.0):

Determines which quantile to optimize for during sampling.

| Value | Meaning | Use Case |
|-------|---------|----------|
| 0.5 | Optimize for median (P50) | Balance between average and tail |
| 0.9 | Optimize for 90th percentile | Good balance (default) |
| 0.95 | Optimize for 95th percentile | Strict SLA requirements |
| 0.99 | Optimize for 99th percentile | Minimize tail latency |

**How it affects selection:**
- Higher quantiles → More conservative selection
- Avoids instances with high tail latency
- Better for SLA-sensitive workloads

### When to Use

**Best for:**
- Optimizing for tail latency (p95, p99)
- Production workloads with strict SLAs
- Workloads with high runtime variability
- When you care about worst-case latency

**Advantages:**
- ✅ Optimizes for specific latency percentiles
- ✅ Handles runtime variability well
- ✅ Balances load probabilistically
- ✅ Adapts to changing conditions

**Disadvantages:**
- ❌ More complex than other strategies
- ❌ Requires accurate quantile predictions
- ❌ Slightly higher computational cost (O(N) sampling)

### Example

```
Instances with quantile distributions:
  A: quantiles=[0.5, 0.9], values=[100, 200]
  B: quantiles=[0.5, 0.9], values=[150, 180]
  C: quantiles=[0.5, 0.9], values=[120, 250]

New task prediction:
  quantiles=[0.5, 0.9], values=[50, 80]

Sampling (target_quantile=0.9):
  A: sample from queue → 180ms, add task → 180 + 70 = 250ms
  B: sample from queue → 170ms, add task → 170 + 65 = 235ms
  C: sample from queue → 200ms, add task → 200 + 75 = 275ms

Selected: Instance B (minimum sampled total time: 235ms)

Note: Results vary across submissions due to probabilistic sampling
```

## Strategy Comparison

### Performance Characteristics

| Strategy | Selection Time | Predictor Calls | Queue Updates | Complexity |
|----------|---------------|-----------------|---------------|------------|
| Round Robin | < 0.1ms | 0 | None | O(1) |
| Minimum Time | < 0.5ms | N (instances) | Error propagation | O(N) |
| Probabilistic | < 1ms | N (instances) | Monte Carlo (1000 samples) | O(N) |

All strategies are suitable for production use.

### Optimization Goals

| Strategy | Optimizes For | Best Metric | Worst Metric |
|----------|--------------|-------------|--------------|
| Round Robin | Fairness | Equal distribution | Average latency, tail latency |
| Minimum Time | Average latency | Mean completion time | Tail latency, fairness |
| Probabilistic | Tail latency | P95/P99 latency | Slight overhead |

### Queue Information Comparison

| Strategy | Queue Type | Fields | Update Method |
|----------|-----------|--------|---------------|
| Round Robin | InstanceQueueProbabilistic | quantiles, values | Not used |
| Minimum Time | InstanceQueueExpectError | expected_time_ms, error_margin_ms | Error propagation |
| Probabilistic | InstanceQueueProbabilistic | quantiles, values | Monte Carlo sampling |

## Implementation Details

### Predictor Integration

All strategies (except round robin) call the predictor service to estimate task runtime:

**Request:**
```python
{
  "model_id": "easyocr/dec",
  "platform_info": {
    "software_name": "docker",
    "software_version": "20.10",
    "hardware_name": "nvidia-rtx-3090"
  },
  "features": {
    "height": 512,
    "width": 512
  },
  "prediction_type": "expect_error"  # or "quantile"
}
```

**Response:**

For **expect_error** (min_time strategy):
```python
{
  "expect": 150.5,
  "error": 25.3
}
```

For **quantile** (probabilistic/round_robin):
```python
{
  "quantiles": [0.5, 0.9, 0.95, 0.99],
  "values": [120.5, 250.3, 300.8, 450.2]
}
```

### Platform Batching

Instances with identical `platform_info` share predictions:

```
Instances:
  A: platform={docker, 20.10, nvidia-rtx-3090}
  B: platform={docker, 20.10, nvidia-rtx-3090}
  C: platform={docker, 20.10, nvidia-rtx-3080}

Predictor calls for a task:
  1. Predict for platform {docker, 20.10, nvidia-rtx-3090} → used for A and B
  2. Predict for platform {docker, 20.10, nvidia-rtx-3080} → used for C

Total: 2 calls instead of 3
```

This optimization reduces latency and predictor load.

## Choosing a Strategy

### Decision Tree

```
Start: Do you have a predictor service?
  ├─ No → Use Round Robin
  └─ Yes
      └─ What is your primary goal?
          ├─ Minimize average latency → Use Minimum Time
          ├─ Meet SLA requirements (p95, p99) → Use Probabilistic
          └─ Simple/testing → Use Round Robin
```

### Recommendation by Use Case

| Use Case | Recommended Strategy | Target Quantile |
|----------|---------------------|-----------------|
| Development/Testing | Round Robin | N/A |
| Batch processing (throughput) | Minimum Time | N/A |
| Real-time API (average latency) | Minimum Time | N/A |
| Real-time API (p95 SLA) | Probabilistic | 0.95 |
| Real-time API (p99 SLA) | Probabilistic | 0.99 |
| Mixed workload | Probabilistic | 0.9 |

### Benchmarking

To compare strategies empirically:

```bash
# Test each strategy with same workload
for strategy in round_robin min_time probabilistic; do
  echo "Testing $strategy..."

  # Switch strategy
  curl -X POST http://localhost:8000/strategy/set \
    -H "Content-Type: application/json" \
    -d "{\"strategy_name\": \"$strategy\"}"

  # Run benchmark
  python benchmark.py --workload standard

  # Collect metrics
  curl "http://localhost:8000/health" | jq .stats
done
```
