# Product Requirements Document: Experiment 10 - Multi-Stage Workflow with Intelligent Redeployment

## Project Overview

Create Experiment 10 based on Experiment 04, implementing a three-stage workflow with fixed fanouts (8, 15, 1) and intelligent instance redeployment between stages. The experiment will compare three scheduling strategies (min_time, round_robin, probabilistic) with and without dynamic redeployment to quantify the performance benefits of adaptive resource allocation.

## Background

Experiment 04 tests workflows with random fanout (3-8 B tasks per A task). Experiment 10 extends this by:
- Using three separate stages with fixed fanouts
- Implementing intelligent instance redeployment between stages
- Minimizing the number of instances that need to be redeployed
- Optimizing instance allocation based on fanout and task time distribution

## Core Requirements

### 1. Three-Stage Workflow Architecture

**Stage 1: A→8B workflows**
- Execute N workflows (e.g., 100)
- Each A task generates exactly 8 B tasks
- All workflows must complete before stage transition

**Stage 2: A→15B workflows**
- Execute M workflows (e.g., 100)
- Each A task generates exactly 15 B tasks
- All workflows must complete before stage transition

**Stage 3: A→1B workflows**
- Execute K workflows (e.g., 100)
- Each A task generates exactly 1 B task
- Final stage

**Stage Transitions**:
- Wait for all workflows in current stage to complete
- Calculate optimal instance allocation for next stage
- Execute intelligent redeployment (only change necessary instances)
- Verify new configuration is active
- Start next stage

### 2. Six Experimental Scenarios

**Group A: Static Configuration (Baseline)**
- Experiment 1: min_time strategy, no redeployment
- Experiment 2: round_robin strategy, no redeployment
- Experiment 3: probabilistic strategy, no redeployment
- Uses single fixed instance allocation for all three stages

**Group B: Dynamic Redeployment**
- Experiment 4: min_time strategy, with redeployment
- Experiment 5: round_robin strategy, with redeployment
- Experiment 6: probabilistic strategy, with redeployment
- Reconfigures instances between stages for optimal performance

### 3. Intelligent Redeployment System

**Redeployment Timing**:
- Triggered after stage completion (not during)
- Only at stage boundaries (2 redeployment events per experiment)

**Redeployment Planning**:
- Calculate optimal A/B instance ratio based on:
  - Next stage's fanout (8, 15, or 1)
  - Historical task completion times from previous stages
  - Queueing theory (M/M/c model considerations)
  - Scheduling strategy characteristics
- Generate minimal change plan:
  - Identify which instances need to change schedulers
  - Minimize total number of redeployments
  - Preserve instances that don't need to change

**Redeployment Execution**:
- Use instance `/model/restart` API
- Change scheduler_url parameter only (model stays as sleep_model)
- Execute redeployments sequentially with status polling
- Wait for all redeployments to complete
- Verify new configuration before starting next stage

**Safety Constraints**:
- Total instance count remains fixed (e.g., 30 instances)
- Only adjust A-group vs B-group allocation
- Ensure graceful drain before restart (handled by restart API)
- Timeout and retry mechanisms for failed redeployments

### 4. Optimal Ratio Calculation

**Input Parameters**:
- Next stage fanout (8, 15, or 1)
- Average A task completion time
- Average B task completion time
- Total number of instances
- Scheduling strategy type

**Calculation Approach**:
```
Base formula:
  optimal_ratio = (avg_a_time / avg_b_time) * (1 / fanout)

With safety factor (to prevent queue buildup):
  num_a = total / (1 + fanout * (avg_b_time / avg_a_time) * safety_factor)
  num_b = total - num_a

  where safety_factor ≈ 1.2 (20% buffer)

Example:
  total=30, fanout=8, avg_a=5s, avg_b=5s
  num_a = 30 / (1 + 8 * 1.0 * 1.2) = 30 / 10.6 ≈ 3
  num_b = 27
```

**Strategy-Specific Adjustments**:
- min_time: Can use calculated ratio directly (efficient load balancing)
- round_robin: May need 10-15% more B instances (less efficient)
- probabilistic: Adjust based on quantile distribution

### 5. Implementation Components

**Component 1: Multistage Workload Generator** (workload_generator.py)
- Function: `generate_multistage_fanout_distribution()`
- Input: num_workflows_per_stage=[N1, N2, N3], fanouts=[8, 15, 1]
- Output: Dict with stage-specific fanout arrays
- Task time distributions remain bimodal (inherit from Exp 04)

**Component 2: Optimal Ratio Calculator** (optimal_ratio_calculator.py)
- Class: `OptimalRatioCalculator`
- Method: `calculate_optimal_ratio(fanout, task_stats, total, strategy)`
- Returns: (num_a_instances, num_b_instances)
- Implements formula with safety factors

**Component 3: Redeployment Manager** (redeployment_manager.py)
- Class: `RedeploymentManager`
- Methods:
  - `get_current_instance_allocation()`: Query current state
  - `plan_redeployment(current, target)`: Generate minimal change plan
  - `execute_redeployment(plan)`: Execute via /model/restart API
- Handles API calls, status polling, error handling

**Component 4: Multistage Test Script** (test_multistage_workflow.py)
- Main function: `run_multistage_experiment(strategy, enable_redeployment)`
- Orchestrates three-stage execution
- Integrates ratio calculator and redeployment manager
- Collects per-stage metrics
- CLI arguments: --strategy, --enable-redeployment, --num-workflows-per-stage

**Component 5: Smart Service Startup Script** (start_all_services.sh)
- Accepts: N_TOTAL (total instances) or N1, N2 (A-group, B-group)
- Auto-calculates initial allocation for stage 1 (fanout=8) if not provided
- Starts: Predictor, Scheduler A, Scheduler B, N1+N2 instances
- Health checks all services

### 6. Metrics and Evaluation

**Per-Stage Metrics**:
- Workflow completion time (A submit → last B complete)
- Task completion times (A and B separately)
- Queue lengths over time
- Instance utilization

**Redeployment Metrics**:
- Number of instances redeployed
- Redeployment duration (drain → restart → register)
- Downtime impact

**Comparison Metrics**:
- Group A (no redeployment) average workflow time per stage
- Group B (with redeployment) average workflow time per stage
- Performance improvement percentage
- Overhead cost of redeployment

**Expected Outcomes**:
- Stage 1 (fanout=8): B-heavy allocation optimal
- Stage 2 (fanout=15): Even more B-heavy allocation optimal
- Stage 3 (fanout=1): More balanced or A-heavy allocation optimal
- Group B should show lower overall workflow times despite redeployment overhead

## Technical Specifications

### API Interfaces

**Instance Restart API** (`POST /model/restart`):
```json
{
  "model_id": "sleep_model",
  "scheduler_url": "http://localhost:8200"
}
```

**Restart Status API** (`GET /model/restart/status?operation_id=<id>`):
Returns status: pending, draining, waiting_tasks, stopping_model, deregistering, starting_model, registering, completed, failed
