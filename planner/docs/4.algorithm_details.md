# Algorithm Details

## Problem Formulation

Given:
- **M** instances (machines) and **N** model types
- A batch capacity matrix **B** of size M x N, where `B[i][j]` is the throughput of instance `i` when running model `j` (0 means incompatible)
- An initial deployment vector **initial** of length M, where `initial[i]` is the model index currently running on instance `i` (or -1 if undeployed)
- A change constraint factor **a** (0 < a <= 1), meaning at most `floor(a * M)` instances may change
- A target request distribution vector **target** of length N

Find a deployment vector **x** of length M (each `x[i]` in `{0, ..., N-1}`) that:
1. Satisfies `B[i][x[i]] > 0` for all `i` (feasibility)
2. Changes at most `floor(a * M)` instances from the initial state
3. Minimizes a chosen objective function measuring how well the resulting service capacity matches the target distribution

## Objective Functions

The Planner supports three objective functions. In all formulas, `C[j]` denotes the total service capacity for model `j` under the current deployment:

```
C[j] = sum(B[i][j] for all i where x[i] == j)
```

### 1. Relative Error (L1 Ratio Distance)

Measures the L1 distance between the capacity ratio and the target ratio:

```
f(x) = sum_j |C[j] / sum(C) - target[j] / sum(target)|
```

This normalizes both capacity and target to probability distributions and measures their total variation distance. Range: [0, 2]. This is the default and the only objective fully supported by integer programming.

### 2. Ratio Difference (L-infinity Proportion Distance)

Measures the L-infinity distance between the capacity proportion and the target proportion:

```
f(x) = max_j |C[j] / sum(C) - target[j] / sum(target)|
```

Normalizes both capacity and target to probability distributions and measures their maximum pointwise deviation. Range: [0, 1]. More sensitive to outliers than relative error (L1).

### 3. Weighted Squared Error

Weighted squared deviation with inverse-target weights:

```
scale = sum(C) / sum(target)
w[j] = 1 / (target[j] + epsilon)
f(x) = sum_j w[j] * (C[j] - scale * target[j])^2
```

Models with low target demand are penalized more heavily for deviations.

## Simulated Annealing

### Pseudocode

```
function SA_Optimize(B, initial, a, target, objective_method):
    deployment = initial.copy()
    if any(initial[i] == -1):
        deployment = generate_initial_deployment()  # assign highest-capacity model per instance
        initial = deployment.copy()

    current_score = objective(deployment, method)
    best = deployment, best_score = current_score
    T = initial_temp

    while T > final_temp and iterations < max_iterations:
        for iter in 1..iterations_per_temp:
            neighbor = random_neighbor(deployment)
            if neighbor is valid:
                delta = objective(neighbor) - current_score
                if delta < 0 or random() < exp(-delta / T):
                    deployment = neighbor
                    current_score = objective(neighbor)
                    if current_score < best_score:
                        best = deployment, best_score = current_score
        T = T * cooling_rate

    return best, best_score, stats
```

### Parameters

| Parameter | Default | Description |
|---|---|---|
| `initial_temp` | 100.0 | Starting temperature |
| `final_temp` | 0.01 | Temperature at which to stop |
| `cooling_rate` | 0.95 | Multiplicative cooling: `T = T * rate` each step |
| `max_iterations` | 5000 | Hard limit on total iterations |
| `iterations_per_temp` | 100 | Iterations before each cooling step |

### Neighbor Generation

At each iteration:
1. Pick a random instance `i` from `{0, ..., M-1}`
2. Find all models `j != x[i]` where `B[i][j] > 0`
3. Pick one uniformly at random
4. Return the new deployment with `x[i] = j`

### Acceptance Criterion (Metropolis)

A neighbor with objective delta `delta` is accepted if:
- `delta < 0` (improvement), or
- `random() < exp(-delta / T)` (probabilistic uphill move)

At high temperatures, worse solutions are accepted frequently (exploration). As temperature drops, only improvements are accepted (exploitation).

## Integer Programming

### MILP Formulation

**Decision variables:**
- `x[i,j]` ∈ {0, 1} — whether instance `i` deploys model `j`
- `y[i]` ∈ {0, 1} — whether instance `i` changed from initial

**Constraints:**

1. **One model per instance:** `sum_j x[i,j] = 1` for all `i`
2. **Capacity feasibility:** `x[i,j] = 0` if `B[i][j] = 0`
3. **Change detection:**
   - If `initial[i] = -1`: `y[i] = 1` (any deployment counts as change)
   - Otherwise: `y[i] >= 1 - x[i, initial[i]]`
4. **Change limit:** `sum_i y[i] <= floor(a * M)`

**Objective (relative_error only):**

Linearized using auxiliary deviation variables `d[j] >= 0`:

```
capacity[j] = sum_i B[i][j] * x[i,j]
total_capacity = sum_j capacity[j]
target_ratio[j] = target[j] / sum(target)

d[j] >= capacity[j] - target_ratio[j] * total_capacity
d[j] >= target_ratio[j] * total_capacity - capacity[j]

minimize sum_j d[j]
```

For other objective methods, the solver falls back to minimizing the number of changes (`sum_i y[i]`) as a linear surrogate, with a warning logged.

### Solver

The solver is PuLP with the CBC backend (`PULP_CBC_CMD`) by default. Other PuLP-compatible solvers can be specified via the `solver_name` parameter.

| Parameter | Default | Description |
|---|---|---|
| `solver_name` | `"PULP_CBC_CMD"` | PuLP solver backend |
| `time_limit` | 300 | Maximum solve time in seconds |

### Important Limitation

Only `relative_error` has a fully linearized formulation. For `ratio_difference` and `weighted_squared`, the IP solver uses change-count minimization as a surrogate objective.

## Algorithm Comparison

| Aspect | Simulated Annealing | Integer Programming |
|---|---|---|
| Optimality | Approximate (stochastic) | Exact (for relative_error) |
| Speed (small M,N) | Fast (<1s) | Fast (<1s) |
| Speed (large M,N) | Scales linearly | Exponential worst-case |
| Objective support | All 3 methods | Only relative_error (others use surrogate) |
| Determinism | Non-deterministic (seeded) | Deterministic |
| Global search | Yes (temperature-based) | Yes (branch-and-bound) |

## PyLet Architecture

The PyLet integration layer translates optimizer decisions into cluster operations through five components:

| Component | Module | Responsibility |
|---|---|---|
| **PyLetClient** | `pylet/client.py` | Wraps `pylet.submit()`, `pylet.get()`, `pylet.cancel()`. Translates model IDs and backends into PyLet commands |
| **InstanceManager** | `pylet/instance_manager.py` | Orchestrates complete instance lifecycle: deploy → health-check → register → drain → terminate |
| **DeploymentExecutor** | `pylet/deployment_executor.py` | Computes state diff (current vs target), generates add/remove actions, executes via InstanceManager |
| **MigrationExecutor** | `pylet/migration_executor.py` | Implements cancel-and-resubmit migration: terminate old → deploy new → register new |
| **SchedulerClient** | `pylet/scheduler_client.py` | HTTP client for scheduler's `/v1/instance/register`, `/v1/instance/drain`, `/v1/instance/remove` |

These are composed by the **PyLetDeploymentService** (`pylet/deployment_service.py`), which is the high-level facade used by the API layer.

### Instance State Machine

```
DEPLOYING → WAITING_HEALTH → REGISTERING → ACTIVE → DRAINING → TERMINATING → TERMINATED
                                              ↓
                                           FAILED
```

| State | Meaning |
|---|---|
| `DEPLOYING` | PyLet instance submitted, waiting for worker allocation |
| `WAITING_HEALTH` | Instance running, waiting for health check endpoint |
| `REGISTERING` | Healthy, registering with scheduler |
| `ACTIVE` | Registered and serving traffic |
| `DRAINING` | Scheduler drain initiated (stop new tasks) |
| `TERMINATING` | Being cancelled in PyLet |
| `TERMINATED` | Fully terminated |
| `FAILED` | Failed at any stage |

## Deployment Flow

When `/v1/deploy` or `/v1/optimize` is called:

1. **Optimize**: Run SA or IP to compute optimal deployment vector
2. **Map to target_state**: Convert deployment array `[0, 1, 1, 2]` + `model_ids ["a","b","c"]` → `{"a": 1, "b": 2, "c": 1}`
3. **Compute diff**: DeploymentExecutor compares current state to target state
4. **Execute removes first**: Terminate excess instances (drain → cancel via PyLet)
5. **Execute adds**: Deploy new instances via `pylet.submit()` with appropriate backend command
6. **Wait for ready**: Poll PyLet until instances reach RUNNING status
7. **Register**: Register each ready instance with the per-model scheduler

## Migration Strategy

The Planner uses **cancel-and-resubmit** since PyLet does not support live migration:

1. **Drain**: Call scheduler's `/v1/instance/drain` to stop new task assignments
2. **Wait**: Poll `/v1/instance/drain/status` until `can_remove=true` or timeout
3. **Remove**: Call scheduler's `/v1/instance/remove`
4. **Cancel**: Call `pylet.cancel(delete=True)` to terminate the old instance
5. **Deploy new**: Call `pylet.submit()` with same model ID and backend
6. **Register new**: Register the new instance with the scheduler

This approach ensures zero task loss at the cost of brief capacity reduction during migration.
