# Introduction

The Planner is a stateless HTTP microservice that computes optimal model-to-instance deployment configurations. Given a set of instances with heterogeneous batch-processing capacities and a target request distribution, it solves a constrained optimization problem to match service capacity to demand. When paired with PyLet, it also executes the computed plan by provisioning and deregistering instances automatically.

- **Planner** exposes optimization and deployment endpoints.
- **Scheduler** receives instance registrations so it can route inference requests per model.
- **PyLet** manages the physical instance lifecycle (submit, health-check, cancel) on a cluster of workers.

## Key Concepts

| Symbol | Meaning |
|---|---|
| **M** | Number of instances (machines/VMs) |
| **N** | Number of model types |
| **B** | Batch capacity matrix [M x N] вҖ” `B[i][j]` = throughput of instance `i` running model `j` |
| **a** | Change constraint factor (0 < a <= 1) вҖ” at most `floor(a * M)` instances may change |
| **target** | Target request distribution [N] вҖ” desired ratio of traffic per model |
| **initial** | Current deployment vector [M] вҖ” `initial[i]` = model index on instance `i`, or -1 if undeployed |
| **deployment** | Output vector [M] вҖ” optimized model assignment per instance |

## Operational Modes

### Standalone (Plan-Only)

Set `PYLET_ENABLED=false` (default). The Planner exposes `POST /v1/plan` which accepts an optimization problem and returns the computed deployment vector, score, and capacity breakdown. No instances are provisioned вҖ” callers apply the plan externally.

### PyLet (Full Cluster Management)

Set `PYLET_ENABLED=true` with `PYLET_HEAD_URL` pointing to a PyLet head node. In addition to `/v1/plan`, the Planner exposes deployment endpoints (`/v1/deploy`, `/v1/deploy_manually`, `/v1/scale`, `/v1/migrate`, `/v1/optimize`, `/v1/terminate-all`) that compute plans and execute them by provisioning real instances via PyLet, then registering them with the per-model scheduler.
