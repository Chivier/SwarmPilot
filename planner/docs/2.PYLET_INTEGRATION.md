# PyLet Integration Guide

This document describes the PyLet integration in the Planner service, which provides automated cluster management for deploying, scaling, and managing model instances.

## Overview

PyLet is a cluster management system that the Planner integrates with for:
- Deploying model instances across worker nodes
- Scaling model counts based on optimizer decisions
- Migrating instances between models
- Managing instance lifecycle (health checks, drain, termination)

## Components

### 1. PyLetClient (`client.py`)

Low-level wrapper around the PyLet API.

**Task ID:** PYLET-007

**Purpose:** Provides clean interface for PyLet operations, handling API communication and error management.

**Key Operations:**
- `deploy_model()` - Deploy new model instances
- `get_instance()` - Get instance status
- `list_instances()` - List all instances
- `cancel_instance()` - Cancel a running instance
- `wait_for_running()` - Wait for instance to be healthy

**Data Structures:**
```python
@dataclass
class InstanceInfo:
    pylet_id: str          # PyLet instance UUID
    model_id: str          # Model identifier
    endpoint: str | None   # HTTP endpoint when running
    status: str            # PyLet status (PENDING, RUNNING, etc.)
    backend: str           # Model backend (vllm, sglang)
    labels: dict           # Additional instance labels
```

---

### 2. InstanceManager (`instance_manager.py`)

Orchestrates complete instance lifecycle.

**Task ID:** PYLET-008

**Purpose:** Manages the full instance lifecycle: deploy via PyLet, wait for health, register with scheduler, and cleanup on termination.

**Key Operations:**
- `deploy_instances()` - Deploy new instances
- `wait_instances_ready()` - Wait for health + register
- `terminate_instances()` - Clean termination
- `drain_instance()` - Prepare for removal

**Instance States:**
```python
class ManagedInstanceStatus(str, Enum):
    DEPLOYING = "deploying"       # Submitted, waiting for running
    WAITING_HEALTH = "waiting_health"  # Running, waiting health check
    REGISTERING = "registering"   # Healthy, registering with scheduler
    ACTIVE = "active"             # Registered and serving
    DRAINING = "draining"         # Stopping new tasks
    TERMINATING = "terminating"   # Being terminated
    TERMINATED = "terminated"     # Fully terminated
    FAILED = "failed"             # Failed to deploy
```

**ManagedInstance Structure:**
```python
@dataclass
class ManagedInstance:
    pylet_id: str                # PyLet instance UUID
    instance_id: str             # Scheduler instance ID
    model_id: str                # Model identifier
    endpoint: str | None         # HTTP endpoint when running
    backend: str                 # Model backend
    status: ManagedInstanceStatus
    gpu_count: int               # GPUs allocated
    worker_id: str | None        # Worker where instance runs
    error: str | None            # Error if failed
    created_at: float            # Creation timestamp
```

### 3. DeploymentExecutor (`deployment_executor.py`)

Translates optimizer output into PyLet operations.

**Task ID:** PYLET-009

**Purpose:** Computes diff between current and target state, then executes add/remove operations.

**Key Operations:**
- `execute()` - Execute deployment plan
- `compute_plan()` - Create deployment plan from state diff

**Data Structures:**
```python
@dataclass
class DeploymentAction:
    action_type: str      # "add" or "remove"
    model_id: str         # Model to add/remove
    count: int            # Number of instances
    target_worker: str | None  # Target worker for adds
    pylet_ids: list[str]  # IDs for removals

@dataclass
class DeploymentPlan:
    actions: list[DeploymentAction]
    total_adds: int
    total_removes: int

@dataclass
class ExecutionResult:
    success: bool
    added_count: int
    removed_count: int
    active_instances: list[ManagedInstance]
    failed_adds: list[tuple[str, str]]     # (model_id, error)
    failed_removes: list[tuple[str, str]]  # (pylet_id, error)
```

---

### 4. MigrationExecutor (`migration_executor.py`)

Handles instance migration between models.

**Task ID:** PYLET-010

**Purpose:** Migrates instances from one model to another via cancel-and-resubmit pattern.

**Key Operations:**
- `migrate_instance()` - Migrate single instance
- `migrate_batch()` - Migrate multiple instances

**Migration Strategy:**
1. Drain the instance (stop accepting new tasks)
2. Wait for pending tasks to complete
3. Cancel the old instance
4. Deploy new instance with target model
5. Register with scheduler

---

### 5. PyLetDeploymentService (`deployment_service.py`)

High-level service for optimizer integration.

**Task ID:** PYLET-013

**Purpose:** Provides simplified interface for the planner to manage model instances via PyLet.

**Key Operations:**
- `apply_deployment()` - Execute target state deployment
- `scale_model()` - Scale specific model
- `migrate_instance()` - Migrate instance to new model
- `get_current_state()` - Get current model counts
- `get_active_instances()` - Get all active instances
- `terminate_all()` - Terminate all instances

**Configuration:**
```python
default_backend: str = "vllm"      # Default model backend
default_gpu_count: int = 1         # Default GPUs per instance
deploy_timeout: float = 300.0      # Deployment timeout (seconds)
drain_timeout: float = 60.0        # Drain timeout (seconds)
```

## API Endpoints

The PyLet API endpoints are defined in `src/pylet_api.py` (PYLET-014).

### GET /status

Get current PyLet service status.

**Response:**
```json
{
  "enabled": true,
  "initialized": true,
  "current_state": {"model-a": 2, "model-b": 1},
  "total_instances": 3,
  "active_instances": [
    {
      "pylet_id": "p1",
      "instance_id": "i1",
      "model_id": "model-a",
      "endpoint": "http://localhost:8001",
      "status": "active",
      "error": null
    }
  ]
}
```

### POST /deploy

Deploy instances to target state.

**Request:**
```json
{
  "target_state": {"model-a": 2, "model-b": 1},
  "wait_for_ready": true,
  "register_with_scheduler": true
}
```

**Response:**
```json
{
  "success": true,
  "added_count": 2,
  "removed_count": 0,
  "active_instances": [...],
  "failed_adds": [],
  "failed_removes": [],
  "error": null
}
```

### POST /scale

Scale a specific model.

**Request:**
```json
{
  "model_id": "model-a",
  "target_count": 3,
  "wait_for_ready": true
}
```

**Response:**
```json
{
  "success": true,
  "model_id": "model-a",
  "previous_count": 2,
  "current_count": 3,
  "added": 1,
  "removed": 0,
  "active_instances": [...],
  "error": null
}
```

### POST /migrate

Migrate an instance to a different model.

**Request:**
```json
{
  "pylet_id": "p1",
  "target_model_id": "model-b"
}
```

**Response:**
```json
{
  "success": true,
  "old_pylet_id": "p1",
  "new_pylet_id": "p2",
  "model_id": "model-b",
  "endpoint": "http://localhost:8002",
  "error": null
}
```

### POST /optimize

Run optimizer and deploy result via PyLet.

**Request:**
```json
{
  "target": [0.5, 0.3, 0.2],
  "model_ids": ["model-a", "model-b", "model-c"],
  "B": [[1.0, 0.8, 0.6], [0.9, 1.0, 0.7]],
  "a": 0.3,
  "objective_method": "ratio_difference",
  "algorithm": "simulated_annealing",
  "wait_for_ready": true
}
```

**Response:**
```json
{
  "deployment": [0, 1],
  "objective_value": 0.05,
  "service_capacity": [1.0, 1.0, 0.0],
  "changes": 1,
  "optimization_stats": {...},
  "deployment_success": true,
  "added_count": 2,
  "removed_count": 0,
  "active_instances": [...],
  "error": null
}
```

### POST /terminate-all

Terminate all PyLet-managed instances.

**Query Parameters:**
- `wait_for_drain`: Wait for instances to drain before termination

**Response:**
```json
{
  "success": true,
  "total": 3,
  "succeeded": 3,
  "failed": 0,
  "details": {"p1": true, "p2": true, "p3": true}
}
```

## Configuration

PyLet integration is configured via environment variables:

| Variable | Default | Description |
|----------|---------|-------------|
| `PYLET_ENABLED` | `false` | Enable PyLet integration |
| `PYLET_HEAD_URL` | - | PyLet head service URL (required if enabled) |
| `PYLET_BACKEND` | `vllm` | Default model backend |
| `PYLET_GPU_COUNT` | `1` | Default GPUs per instance |
| `PYLET_DEPLOY_TIMEOUT` | `300.0` | Deployment timeout (seconds) |
| `PYLET_DRAIN_TIMEOUT` | `60.0` | Drain timeout (seconds) |

## Usage Examples

### Basic Deployment
```python
from src.pylet.deployment_service import PyLetDeploymentService

service = PyLetDeploymentService(
    pylet_head_url="http://pylet-head:8000",
    scheduler_url="http://scheduler:8001",
)
service.init()

# Deploy to target state
result = service.apply_deployment(
    target_state={"model-a": 2, "model-b": 1},
)
print(f"Deployed: {result.total_added} instances")
```

### Scaling
```python
# Scale a specific model
result = service.scale_model("model-a", target_count=5)
print(f"Model-a now has {result.current_count} instances")
```

### Migration
```python
# Migrate instance to different model
result = service.migrate_instance("p1", "model-c")
print(f"Migrated to: {result.new_pylet_id}")
```

### With Optimizer
```python
from src.core.swarm_optimizer import SimulatedAnnealingOptimizer

optimizer = SimulatedAnnealingOptimizer()
plan = optimizer.optimize(M=4, N=3, B=batch_matrix, target=target_dist)

# Convert optimizer output to deployment
model_ids = ["model-a", "model-b", "model-c"]
target_state = {}
for i, model_idx in enumerate(plan.deployment):
    if model_idx >= 0:
        model_id = model_ids[model_idx]
        target_state[model_id] = target_state.get(model_id, 0) + 1

result = service.apply_deployment(target_state=target_state)
```

## Error Handling

### Partial Failures
The deployment system handles partial failures gracefully:
- Successful deployments continue even if some fail
- Failed operations are tracked in `failed_adds` / `failed_removes`
- The `success` flag is `True` only if all operations succeed

### Retry Logic
- PyLet client includes automatic retry for transient errors
- Exponential backoff with configurable limits
- Health checks retry until timeout

### Cleanup on Failure
- Failed instances are automatically cleaned up
- Partial deployments can be rolled back via `terminate_all()`
