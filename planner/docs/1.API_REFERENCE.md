# Planner API Reference

## Overview

The Planner service optimizes model deployment across instances using configurable algorithms (simulated annealing or integer programming). It automatically maps model names to IDs, computes optimal assignments, and executes deployments.

**Key Capabilities:**
- Optimal model-to-instance assignment with configurable algorithms
- Constraint-aware planning (batch capacity, change limits)
- Multi-objective optimization functions
- Automatic model name ↔ ID mapping
- Automated deployment execution


## Data Models

### PlannerInput

Optimization parameters and algorithm configuration.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| **Core Parameters** ||||
| `M` | `int` | *required* | Number of instances (> 0) |
| `N` | `int` | *required* | Number of model types (> 0) |
| `B` | `List[List[float]]` | *required* | Batch capacity matrix [M×N] |
| `initial` | `List[int]` | *required* | Initial deployment [M], -1 = no model |
| `a` | `float` | *required* | Change constraint (0 < a ≤ 1) |
| `target` | `List[float]` | *required* | Target request distribution [N] |
| **Algorithm** ||||
| `algorithm` | `str` | `"simulated_annealing"` | `"simulated_annealing"` or `"integer_programming"` |
| `objective_method` | `str` | `"relative_error"` | `"relative_error"`, `"ratio_difference"`, or `"weighted_squared"` |
| `verbose` | `bool` | `true` | Enable logging |
| **Simulated Annealing** ||||
| `initial_temp` | `float` | `100.0` | Starting temperature |
| `final_temp` | `float` | `0.01` | Ending temperature |
| `cooling_rate` | `float` | `0.95` | Temperature decay |
| `max_iterations` | `int` | `5000` | Max iterations |
| `iterations_per_temp` | `int` | `100` | Iterations per temperature |
| **Integer Programming** ||||
| `solver_name` | `str` | `"PULP_CBC_CMD"` | Solver backend |
| `time_limit` | `int` | `300` | Timeout (seconds) |

### PlannerOutput

| Field | Type | Description |
|-------|------|-------------|
| `deployment` | `List[int]` | Optimized assignment [M] (model IDs) |
| `score` | `float` | Objective value (lower = better) |
| `stats` | `dict` | Algorithm statistics |
| `service_capacity` | `List[float]` | Capacity per model [N] |
| `changes_count` | `int` | Changes from initial state |

### InstanceInfo

| Field | Type | Description |
|-------|------|-------------|
| `endpoint` | `str` | Instance API endpoint |
| `current_model` | `str` | Current model name |

### DeploymentInput

| Field | Type | Description |
|-------|------|-------------|
| `instances` | `List[InstanceInfo]` | Target instances |
| `planner_input` | `PlannerInput` | Optimization config |

### DeploymentOutput

Extends `PlannerOutput` with deployment execution status.

| Field | Type | Description |
|-------|------|-------------|
| *All fields from `PlannerOutput`* |||
| `deployment_status` | `List[DeploymentStatus]` | Per-instance results |
| `success` | `bool` | Overall success flag |
| `failed_instances` | `List[int]` | Failed instance indices |

### DeploymentStatus

| Field | Type | Description |
|-------|------|-------------|
| `instance_index` | `int` | Instance index |
| `endpoint` | `str` | Instance endpoint |
| `target_model` | `str` | Target model name |
| `previous_model` | `str \| null` | Previous model name |
| `success` | `bool` | Success flag |
| `error_message` | `str \| null` | Error details |
| `deployment_time` | `float` | Deployment duration (sec) |

## API Endpoints

### `POST /plan`

Compute optimal deployment plan (no execution).

**Request:** `PlannerInput` | **Response:** `PlannerOutput`

<details>
<summary><b>Example</b></summary>

```bash
curl -X POST http://localhost:8000/plan \
  -H "Content-Type: application/json" \
  -d '{
    "M": 4, "N": 3,
    "B": [[10, 5, 0], [8, 6, 4], [0, 10, 8], [6, 0, 12]],
    "initial": [0, 1, 2, 2],
    "a": 0.5,
    "target": [20, 30, 25]
  }'
```

```json
{
  "deployment": [0, 1, 1, 2],
  "score": 0.0667,
  "stats": {"algorithm": "simulated_annealing", "iterations": 5000, "acceptance_rate": 0.247},
  "service_capacity": [10.0, 16.0, 12.0],
  "changes_count": 1
}
```
</details>

### `POST /deploy`

Compute plan and execute deployment across instances.

**Request:** `DeploymentInput` | **Response:** `DeploymentOutput`

**Workflow:**
1. Extract model names from instances
2. Map model names → IDs (populate `initial` field)
3. Run optimization algorithm
4. Map result IDs → model names
5. Deploy to instances
6. Return results with status

## Algorithm Selection

| Algorithm | Best For | Trade-off | Key Parameters |
|-----------|----------|-----------|----------------|
| **Simulated Annealing** | Large search spaces, fast approximation | Speed ↔ Optimality | Temperature schedule, iterations |
| **Integer Programming** | Guaranteed optimal, smaller problems | Optimality ↔ Time | Solver, time limit |

## Objective Functions

| Method | Formula Focus | Use Case |
|--------|---------------|----------|
| `relative_error` | Relative deviation | Balanced general optimization |
| `ratio_difference` | Ratio mismatch | Proportional fairness |
| `weighted_squared` | Squared errors | Penalize large deviations |

## Environment Variables

### Basic Configuration

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `PLANNER_HOST` | `str` | `"0.0.0.0"` | Server bind address |
| `PLANNER_PORT` | `int` | `8000` | Server port |
| `SCHEDULER_URL` | `str` | `None` | Scheduler service URL for strategy updates |
| `INSTANCE_TIMEOUT` | `int` | `30` | Instance deployment timeout (seconds) |
| `INSTANCE_MAX_RETRIES` | `int` | `3` | Max retry attempts for deployment |
| `INSTANCE_RETRY_DELAY` | `float` | `1.0` | Delay between retries (seconds) |

### Auto-Optimization Configuration

The planner can automatically trigger migration optimization when schedulers report their queue lengths.

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `AUTO_OPTIMIZE_ENABLED` | `bool` | `false` | Enable automatic optimization loop |
| `AUTO_OPTIMIZE_INTERVAL` | `float` | `60.0` | Interval between optimization checks (seconds) |
| `AUTO_OPTIMIZE_COOLDOWN` | `float` | `5.0` | Minimum time after last target update before triggering (seconds) |

**Example:**
```bash
# Enable auto-optimization with 30-second interval
export AUTO_OPTIMIZE_ENABLED=true
export AUTO_OPTIMIZE_INTERVAL=30.0
export AUTO_OPTIMIZE_COOLDOWN=5.0
```

**Auto-Optimization Workflow:**
1. Schedulers periodically report queue lengths via `/submit_target`
2. Planner accumulates targets for all models in the mapping
3. When all models have reported and cooldown period has passed:
   - Triggers `/deploy/migration` with accumulated targets
   - Resets submitted models tracking
4. Loop continues at configured interval

## Additional API Endpoints

### `POST /submit_target`

Submit queue length information from a scheduler. Used by schedulers to report their current task backlog for auto-optimization.

**Request Body:**
```json
{
  "model_id": "model_name",
  "value": 15.0
}
```

| Field | Type | Description |
|-------|------|-------------|
| `model_id` | `str` | Model identifier (matches names from `/deploy`) |
| `value` | `float` | Current queue length (uncompleted tasks) |

**Response:**
```json
{
  "success": true,
  "message": "Target updated for model model_name at index 0",
  "current_target": [15.0, 0.0, 0.0]
}
```

**Notes:**
- Only effective after `/deploy` or `/deploy/migration` has been called
- Model ID is mapped to internal index using the mapping from deployment
- When auto-optimization is enabled and all models have reported, triggers optimization

### `GET /target`

Get the current accumulated target distribution.

**Response:**
```json
{
  "target": [15.0, 20.0, 10.0],
  "submitted_models": ["model_0", "model_1", "model_2"]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `target` | `List[float]` | Current target values per model |
| `submitted_models` | `List[str]` | Models that have submitted values |

Returns `null` target if no deployment has been executed yet.

### `POST /deploy/migration`

Execute migration optimization using current target distribution. Similar to `/deploy` but uses the accumulated target from `/submit_target`.

**Request:** Same as `/deploy`

**Additional Behavior:**
- Uses `_current_target` if available (from `/submit_target` submissions)
- Falls back to `planner_input.target` if no targets submitted
- Resets submitted models tracking after successful deployment
