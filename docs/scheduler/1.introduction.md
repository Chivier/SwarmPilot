# Introduction

The Scheduler is a prediction-based task distribution service that routes inference requests to optimal worker instances. Given a set of registered instances with heterogeneous hardware, the Scheduler uses a built-in predictor to estimate task execution times and a configurable scheduling algorithm to select the instance that minimizes total completion time. It also provides a transparent proxy that forwards arbitrary HTTP requests through the scheduling pipeline, a WebSocket interface for real-time result delivery, and automatic planner integration for cluster-wide coordination.

## Architecture

```
                      ┌─────────────┐
                      │   Client    │
                      └──────┬──────┘
                             │ HTTP / WebSocket
                      ┌──────▼──────┐
                      │  Scheduler  │  /v1/task/submit  → schedule + enqueue
                      │  (FastAPI)  │  /{path}          → transparent proxy
                      └──┬───┬───┬──┘
                         │   │   │
            ┌────────────┘   │   └────────────┐
            │                │                │
     ┌──────▼──────┐  ┌─────▼──────┐  ┌──────▼──────┐
     │  Predictor  │  │  Worker    │  │   Planner   │
     │  (library)  │  │  Instances │  │  (optional) │
     └─────────────┘  └────────────┘  └─────────────┘
```

- **Scheduler** receives tasks, predicts execution times, selects instances via a scheduling algorithm, and enqueues tasks to per-instance worker queue threads.
- **Predictor** runs as an in-process library (not a separate service). It loads trained models from disk and provides quantile or expect-error predictions.
- **Worker Instances** are registered HTTP endpoints that execute model inference. Each gets a dedicated `WorkerQueueThread` that sends tasks sequentially.
- **Planner** (optional) manages cluster-level deployment. The Scheduler can register itself with the Planner on startup and report throughput periodically.

## Key Concepts

| Concept | Meaning |
|---|---|
| **Task** | A unit of work with `task_id`, `model_id`, `task_input`, and `metadata`. Submitted via `/v1/task/submit` or the transparent proxy |
| **Instance** | A registered worker endpoint identified by `instance_id`, `model_id`, `endpoint`, and `platform_info` (software_name, software_version, hardware_name) |
| **Queue Info (Probabilistic)** | Per-instance queue state represented as a quantile CDF: `quantiles` (e.g., [0.5, 0.9, 0.95, 0.99]) and corresponding `values` (ms). Updated via Monte Carlo sampling |
| **Queue Info (ExpectError)** | Per-instance queue state represented as `expected_time_ms` and `error_margin_ms`. Updated via additive accumulation with sqrt error propagation |
| **Prediction** | Estimated execution time for a task on an instance. Produced by the in-process predictor from trained models, using task metadata and platform info |
| **Strategy** | The algorithm used to select which instance receives a task. Seven strategies available: `adaptive_bootstrap` (default), `probabilistic`, `min_time`, `po2`, `round_robin`, `random`, `serverless` |
| **Work Stealing** | When a new instance registers, the Scheduler steals pending tasks from existing instances to balance load across the cluster |

## Operational Modes

### Standalone (No Planner)

The Scheduler operates independently. Instances are registered via `/v1/instance/register`, tasks are submitted via `/v1/task/submit` or the transparent proxy, and results are retrieved via `/v1/task/info` or WebSocket. No external coordination is needed.

### Integrated (With Planner)

When `PLANNER_REGISTRATION_URL`, `SCHEDULER_MODEL_ID`, and `SCHEDULER_SELF_URL` are set, the Scheduler registers itself with the Planner on startup and deregisters on shutdown. When `PLANNER_URL` and `SCHEDULER_AUTO_REPORT` are set, it periodically reports throughput statistics to the Planner. This enables the Planner to make cluster-wide deployment decisions based on real-time scheduler performance.
