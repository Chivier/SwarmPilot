# Algorithm Details

## Problem Formulation

Given:
- A set of training samples, each containing numeric **features** (e.g., `batch_size`, `sequence_length`) and a measured **runtime_ms**
- A **prediction_type** specifying the output format (`expect_error` or `quantile`)

Learn a function `f(features) → prediction` that predicts runtime for unseen feature combinations. The predictor must:
1. Generalize across the feature space from limited training data
2. Provide uncertainty information (error margin or quantile distribution)
3. Handle heterogeneous hardware by keying models on `platform_info`

## MLP Architecture

Both prediction types use the same underlying MLP (multi-layer perceptron) architecture:

```
Input → [Linear → ReLU] × H → Linear → Output
```

| Component | Description |
|---|---|
| **Input layer** | Dimensionality equals number of features (after preprocessing) |
| **Hidden layers** | Configurable list of layer sizes (default: `[64, 32]`) with ReLU activation |
| **Output layer** | Linear output (1 neuron for expect_error, N neurons for N quantiles) |
| **Optimizer** | Adam with configurable learning rate (default: 0.01) |
| **Determinism** | `torch.use_deterministic_algorithms(True)` for reproducibility |

### Configuration

| Parameter | Default | Description |
|---|---|---|
| `epochs` | 500 | Number of training epochs |
| `learning_rate` | 0.01 | Adam optimizer learning rate |
| `hidden_layers` | `[64, 32]` | List of hidden layer sizes |

## Feature Processing

### Feature Extraction

Features are extracted from training samples by separating `runtime_ms` (the target) from all other numeric keys. Feature names are **sorted alphabetically** to ensure consistent ordering across training and prediction.

### Constant Feature Filtering

Features with zero variance across training samples are detected and removed. This prevents numerical issues during normalization (division by near-zero std). During prediction, constant features are automatically filtered — users can provide the full feature set without manual removal.

### Z-Score Normalization

All features are normalized using z-score standardization:

```
X_normalized = (X - mean) / (std + 1e-8)
```

Mean and standard deviation are computed from training data and stored with the model. The same normalization is applied during prediction.

### GPU Feature Enrichment

When `platform_info.hardware_name` matches a known NVIDIA GPU (V100, A100, H100, etc.), hardware specifications are automatically appended to the feature set during prediction:

| Feature | Description |
|---|---|
| `cuda_cores` | Number of CUDA cores |
| `tensor_cores` | Number of Tensor cores |
| `fp32_tflops` | FP32 throughput |
| `fp16_tflops` | FP16 throughput |
| `tensor_tflops` | Tensor core throughput |
| `memory_gb` | GPU memory |
| `memory_bandwidth_gb_s` | Memory bandwidth |

## Expect Error Predictor

### Training

1. Extract features `X` and labels `y` (runtime_ms) from training data
2. Normalize `X` using z-score normalization
3. Train MLP with **MSE loss** (mean squared error):

```
L = (1/N) * sum_i (f(x_i) - y_i)^2
```

4. After training, compute error margin from residuals:

```
error_margin = mean(|y_i - f(x_i)|)  for all training samples i
```

### Prediction Output

```
{
    "expected_runtime_ms": f(x),
    "error_margin_ms": mean_absolute_residual
}
```

The error margin is a fixed value computed once during training. It represents the average prediction error on the training set.

## Quantile Predictor

### Base + Delta Architecture

The quantile predictor uses a specialized architecture that **guarantees monotonicity** between quantile levels by construction, without post-processing.

```
Input → MLP → [base, δ₁, δ₂, ..., δₙ₋₁] → BaseDeltaTransform → [q₀, q₁, ..., qₙ₋₁]
```

The `BaseDeltaTransform` converts raw MLP outputs to monotonic quantiles:

```
q₀ = base
q₁ = base + softplus(δ₁)
q₂ = base + softplus(δ₁) + softplus(δ₂)
...
qᵢ = qᵢ₋₁ + softplus(δᵢ)
```

Since `softplus(x) = log(1 + exp(x)) > 0` for all `x`, each quantile is guaranteed to be greater than or equal to the previous one.

### Pinball Loss

Training uses the pinball loss (quantile loss) function:

```
L_q(y, ŷ) = max(q * (y - ŷ), (q - 1) * (y - ŷ))
```

For each quantile level `q`:
- When `y > ŷ` (underprediction): loss is `q * (y - ŷ)` — penalizes more for higher quantiles
- When `y < ŷ` (overprediction): loss is `(1 - q) * (ŷ - y)` — penalizes more for lower quantiles

The total loss is the mean pinball loss across all quantile levels and samples. An optional monotonicity penalty can be added, though the base+delta architecture typically makes this unnecessary.

### Target Normalization

Unlike expect_error, the quantile predictor also normalizes the target variable (runtime_ms):

```
y_normalized = (y - target_mean) / (target_std + 1e-8)
```

Predictions are denormalized back to the original scale after inference.

### Configuration

| Parameter | Default | Description |
|---|---|---|
| `quantiles` | `[0.5, 0.9, 0.95, 0.99]` | Quantile levels to predict (0 < q < 1) |
| `monotonicity_penalty` | `0.0` | Weight for additional monotonicity enforcement |
| `delta_scale` | `1.0` | Scale factor for delta outputs (higher = larger gaps between quantiles) |

### Prediction Output

```
{
    "quantiles": {
        "0.5":  120.0,   // median runtime
        "0.9":  150.0,   // 90th percentile
        "0.95": 180.0,   // 95th percentile
        "0.99": 250.0    // 99th percentile
    }
}
```

## Data Augmentation

When each feature combination has only a single runtime observation, the MLP cannot learn variance patterns. Data augmentation generates synthetic samples to address this.

### Strategy

For each original sample with runtime `r`:
1. Generate `samples_per_point - 1` synthetic samples (default: 4 additional)
2. Keep the same features, sample new runtime from a distribution centered on `r`

### Coefficient of Variation (CV) Estimation

If the user does not specify a CV, it is auto-calculated:

1. **Group samples** by feature signature (excluding `runtime_ms`)
2. **Multi-sample groups**: Compute per-group CV, take weighted average
3. **Single-sample groups**: Use overall runtime distribution scaled by 0.4
4. **Bound** the result to `[0.1, 1.5]`

### Distributions

**Lognormal** (default):
```
σ_ln = sqrt(log(1 + cv²))
μ_ln = log(r) - σ_ln² / 2
new_runtime ~ Lognormal(μ_ln, σ_ln)
```

**Normal**:
```
σ = r * cv
new_runtime ~ max(Normal(r, σ), 1.0)
```

Lognormal is preferred because runtime distributions are typically right-skewed and strictly positive.

### Configuration

| Parameter | Default | Description |
|---|---|---|
| `enabled` | `true` | Enable data augmentation |
| `cv` | auto | Coefficient of variation (auto-calculated from data if omitted) |
| `samples_per_point` | `5` | Total samples per original (1 original + 4 synthetic) |
| `distribution` | `"lognormal"` | `"lognormal"` or `"normal"` |

## Log Transform

For right-skewed runtime distributions (common in latency data), a log transform can be applied:

1. **Training**: `y = log(runtime_ms)`, normalize, then train on log-space values
2. **Prediction**: denormalize, then `runtime_ms = exp(y)` to restore original scale

This stabilizes training when runtime values span several orders of magnitude.

---

## Residual Calibration

Post-training step that analyzes the ratio between actual and predicted runtimes to estimate uncertainty distribution parameters.

### Procedure

1. Compute median predictions for all training samples
2. Calculate residual ratios: `residual = actual / predicted`
3. Fit log-normal distribution to residuals: `log(residual) ~ N(μ, σ)`
4. Store `(μ, σ)` for use in calibrated quantile estimation

### Parameters

| Parameter | Default | Description |
|---|---|---|
| `enabled` | `false` | Enable residual calibration |
| `min_sigma` | `0.1` | Minimum sigma to prevent degenerate distributions |

## Preprocessing System

The predictor supports composable feature preprocessing pipelines applied before training and prediction.

### Built-in Preprocessors

| Preprocessor | Purpose |
|---|---|
| `MultiplyPreprocessor` | Multiply two feature values to create a derived feature |
| `RemoveFeaturePreprocessor` | Remove specified features from the feature set |
| `TokenLengthPreprocessor` | Count tokens in a text feature |

### Pipeline

Preprocessors are specified via `enable_preprocessors` (list of preprocessor names) and `preprocessor_mappings` (which features each preprocessor operates on). Each preprocessor:

1. Takes target feature values as input
2. Returns processed features and a flag indicating whether to remove originals
3. Is applied in order to both training data and prediction features

### V2 Chain API

The library API provides a composable `PreprocessorChainV2`:

```python
chain = PreprocessorChainV2()
chain.add(MultiplyPreprocessor("compute_cost"))
chain.add(RemoveFeaturePreprocessor())
features = chain.transform(raw_features)
```

## Experiment Mode

Generates synthetic predictions for testing without trained models.

### Activation Conditions

Experiment mode activates when either:
- Features contain an `exp_runtime` field, or
- All `platform_info` fields (`software_name`, `software_version`, `hardware_name`) are set to `"exp"`

### Expect Error Output

```
expected_runtime_ms = exp_runtime
error_margin_ms     = exp_runtime * 0.05
```

### Quantile Output

Generates quantile values from a normal distribution around `exp_runtime`:

```
quantile(p) = exp_runtime + exp_runtime * cv * z_p
```

where `z_p` is the standard normal quantile for probability `p` and `cv` is a fixed coefficient of variation.

## Predictor Comparison

| Aspect | Expect Error | Quantile |
|---|---|---|
| Loss function | MSE | Pinball (quantile loss) |
| Output | Point estimate + error margin | Multiple percentile values |
| Architecture | MLP → 1 output | MLP → base+delta → N outputs |
| Monotonicity | N/A (single output) | Guaranteed by construction |
| Target normalization | None | Z-score on runtime_ms |
| Data augmentation | Not applied | Enabled by default |
| Use case | Simple scheduling, monitoring | SLA-aware scheduling, risk management |

## Model Lifecycle

### Training Flow

1. Receive training request with features and runtimes
2. Apply preprocessors (if configured)
3. Extract feature matrix `X` and label vector `y`
4. Filter constant features
5. (Quantile only) Augment data, apply log transform
6. Normalize features (and targets for quantile)
7. Train MLP with appropriate loss function
8. (Quantile only) Compute residual calibration
9. Serialize model state (architecture + weights + normalization params)
10. Save to storage, invalidate cache

### Prediction Flow

1. Check experiment mode → return synthetic result if active
2. Look up model in cache (LRU) → load from storage on miss
3. Validate prediction type matches stored model
4. Apply preprocessors (if configured)
5. Enrich features with GPU specs (if hardware detected)
6. Filter to model's expected features
7. Normalize features using stored mean/std
8. Run MLP forward pass
9. Denormalize output (and apply exp() if log transform was used)
10. Format and return result
